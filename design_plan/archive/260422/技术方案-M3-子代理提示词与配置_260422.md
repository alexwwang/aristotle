## 一、模块概述

本模块实现 GEAR 编排架构中 Layer 3（子代理）的 prompt 模板系统，包含三个模板常量和一个配置变更：

| 模板/配置 | 用途 | 触发时机 |
|-----------|------|---------|
| `REFLECTOR_PROMPT_TEMPLATE` | R 子代理启动 prompt | `orchestrate_start("reflect")` 返回 `fire_sub(R)` |
| `CHECKER_PROMPT_TEMPLATE` | C 子代理启动 prompt | R 完成 → `orchestrate_on_event("subagent_done")` 返回 `fire_sub(C)` |
| `REVISE_PROMPT_TEMPLATE` | O 修改规则 prompt | `orchestrate_review_action("revise")` 返回 `fire_o` |
| `SKILL_DIR` (config.py) | 协议文件路径根 | 所有 prompt 模板引用 |

设计原则：
1. 模板为 Python format string（`{variable}` 占位符），运行时由 `_build_*_prompt()` 辅助函数填充
2. 每个模板不包含协议逻辑，仅传递参数 + 引用协议文件路径
3. 协议文件（REFLECTOR.md / CHECKER.md）由子代理通过 Read 工具自行加载

## 三、REFLECTOR_PROMPT_TEMPLATE

### 3.1 模板设计

```python
REFLECTOR_PROMPT_TEMPLATE = """\
You are Aristotle's Reflector subagent. Read and execute the full protocol at
{skill_dir}/REFLECTOR.md (read the file first, then follow it step by step).

TARGET_SESSION_ID: {target_session_id}
PROJECT_DIRECTORY: {project_directory}
USER_LANGUAGE: {user_language}
FOCUS_HINT: {focus_hint}
DRAFT_SEQUENCE: {sequence}

Your output is NOT shown to the user. The Coordinator reads your session and \
extracts the DRAFT. Follow REFLECTOR.md exactly — especially STEP R5 to persist \
the DRAFT via persist_draft(sequence={sequence}, content=...).
"""
```

### 3.2 变量说明

| 变量 | 类型 | 来源 | 用途 |
|------|------|------|------|
| `{skill_dir}` | `str` | `config.SKILL_DIR` | 子代理用 Read 工具加载 `REFLECTOR.md` |
| `{target_session_id}` | `str` | SKILL.md 预解析（§3.1.4） | R1 读取目标 session |
| `{project_directory}` | `str` | SKILL.md 从 `cwd` 获取 | 项目级规则 scope 判断 |
| `{user_language}` | `str` | SKILL.md 从近期消息推断 | R2 中英错误模式检测 |
| `{focus_hint}` | `str` | 用户 `--focus` 参数（默认 `"last"`） | R1a 决定读取策略 |
| `{sequence}` | `int` | MCP `_next_sequence()` | R5 调用 `persist_draft` 的序号 |

### 3.3 与 REFLECTOR.md 的关系

模板**不嵌入**协议内容。它只做两件事：

1. **传参** — 将 SESSION PARAMETERS（REFLECTOR.md §行 12-19）的值注入 prompt
2. **引导加载** — 告诉子代理先读 `{skill_dir}/REFLECTOR.md`，然后按步骤执行

这确保了：
- REFLECTOR.md 更新时无需同步修改模板
- prompt 保持简短（~15 行），不浪费子代理 context budget
- 协议文件只在子代理 session 中可见，主 session 永远不加载

### 3.4 构建函数

```python
def _build_reflector_prompt(
    target_session_id: str,
    focus_hint: str,
    sequence: int,
    project_directory: str = "",
    user_language: str = "en-US",
) -> str:
    safe_focus = focus_hint[:200]  # [§7.1] 截断防注入
    return REFLECTOR_PROMPT_TEMPLATE.format(
        skill_dir=str(SKILL_DIR),
        target_session_id=target_session_id,
        project_directory=project_directory,
        user_language=user_language,
        focus_hint=safe_focus,
        sequence=sequence,
    )
```

## 五、REVISE_PROMPT_TEMPLATE

### 5.1 模板设计

```python
REVISE_PROMPT_TEMPLATE = """\
You are revising an Aristotle rule based on user feedback.

ORIGINAL RULE FILE: {rule_path}
ORIGINAL RULE CONTENT:
```
{original_content}
```

USER FEEDBACK: {feedback}

DRAFT CONTEXT:
```
{draft_summary}
```

Write the revised rule. Your output MUST follow this EXACT format:
Line 1: FILE: {rule_path}
Line 2+: The complete revised rule in YAML frontmatter + Markdown body format.
Do NOT include any other text, explanation, or commentary.
"""
```

### 5.2 变量说明

| 变量 | 类型 | 来源 | 用途 |
|------|------|------|------|
| `{rule_path}` | `str` | `workflow["displayed_rules"][rule_index]` | O 知道修改哪个文件 |
| `{original_content}` | `str` | `read_rules` 读取当前文件内容 | O 看到原始规则全文 |
| `{feedback}` | `str` | 用户输入（`orchestrate_review_action` 的 `feedback` 参数） | O 理解修改意图 |
| `{draft_summary}` | `str` | workflow state 中缓存的 DRAFT 摘要（Scan Context + Draft Rules Summary 表格） | O 理解规则产生的上下文 |

### 5.3 输出格式约束

O 的输出必须是严格的两段式格式：

```
FILE: user/2026-04-22_hallucination.md
---
id: "rec_1712743800"
status: "pending"
...frontmatter...
---

## Context
...
## Rule
...
```

解析函数 `_parse_revised_rule` 按 `FILE:` 前缀拆分：
- 第一行提取 `rule_path`
- 第二行起为 `rule_content`

若解析失败，返回 `("⚠️ Could not parse revised rule from O output.")` 给用户。

### 5.4 构建函数

> **注意：** `_build_revise_prompt` 必须对 `original_content`、`feedback`、`draft_summary` 进行花括号转义（§7.3），否则 `str.format()` 遇到规则内容中的 `{}` 会抛出 `KeyError`。以下为唯一的规范实现：

```python
def _build_revise_prompt(
    rule_path: str,
    original_content: str,
    feedback: str,
    draft_summary: str,
) -> str:
    _esc = lambda s: s.replace("{", "{{").replace("}", "}}")
    safe_feedback = feedback[:2000]  # [§7.1] 截断防注入
    return REVISE_PROMPT_TEMPLATE.format(
        rule_path=rule_path,
        original_content=_esc(original_content),
        feedback=_esc(safe_feedback),
        draft_summary=_esc(draft_summary),
    )
```

### 5.5 draft_summary 的获取

`draft_summary` 在 `orchestrate_start("review")` 时从 DRAFT 文件提取，缓存在 workflow state 中。提取逻辑为截取 DRAFT 内容的前 1500 字符（与 M1 `orchestrate_start("review")` 中 `revise` action 的截取逻辑一致），避免将完整 DRAFT 注入 O 的 prompt。

```python
# 在 orchestrate_start("review") 中（revise action 内）：
draft_summary = draft_content[:1500] if draft_content else ""
# 注意：M1 的 revise action 直接从 DRAFT 文件截取前 1500 字符，
# 不使用单独的 _extract_draft_summary 函数。
# workflow state 中不单独缓存 draft_summary。
```

## 七、安全性考虑

### 7.1 Prompt 注入防护

所有模板变量在使用前必须经过安全处理：

| 变量 | 风险 | 防护 |
|------|------|------|
| `feedback`（用户输入） | 用户可能注入指令覆盖协议 | 1. 模板中 `feedback` 位于结构化段落内，有明确的边界标记；2. `_build_revise_prompt` 截断至 2000 字符 |
| `focus_hint` | 用户可能传入恶意文本 | `_build_reflector_prompt` 截断至 200 字符；且该值仅作为 R1a 的策略选择，不直接执行 |
| `original_content` | 规则内容可能包含反向指令 | 不做额外处理——规则内容由系统生成，非用户输入 |
| `draft_summary` | DRAFT 包含用户消息引用 | 外层截取前 1500 字符（与 M1 一致）；且 O 输出必须匹配严格格式，自由文本不会被执行 |

### 7.2 路径安全

- `SKILL_DIR` 通过 `resolve()` 规范化，防止路径遍历
- `draft_file` 路径由 MCP 内部计算（非用户输入），无遍历风险
- `rule_path` 来自 `displayed_rules`（由 `list_rules` 返回），已通过 MCP 层路径校验

### 7.3 格式字符串安全

模板使用 `str.format()`，Python 的 format 不执行任意代码。变量值中的 `{` 和 `}` 会触发 `KeyError` 或 `IndexError`，因此：

- `_build_revise_prompt` 中 `original_content` 可能包含 `{}` — 需在传入前转义：`original_content.replace("{", "{{").replace("}", "}}")`
- 同理，`draft_summary` 和 `feedback` 也需要转义

**修订后的 `_build_revise_prompt`：**

```python
def _build_revise_prompt(
    rule_path: str,
    original_content: str,
    feedback: str,
    draft_summary: str,
) -> str:
    _esc = lambda s: s.replace("{", "{{").replace("}", "}}")
    return REVISE_PROMPT_TEMPLATE.format(
        rule_path=rule_path,
        original_content=_esc(original_content),
        feedback=_esc(feedback),
        draft_summary=_esc(draft_summary),
    )
```

`_build_reflector_prompt` 和 `_build_checker_prompt` 的变量均为系统生成（session_id, path, enum），不含用户自由文本，无需转义。

### 7.4 Context Budget

| 模板 | 行数 | 估计 token 数 | 备注 |
|------|------|-------------|------|
| REFLECTOR | ~15 | ~200 | 加上 REFLECTOR.md (~215 行 ≈ 3000 token)，总计 ~3200 |
| CHECKER | ~10 | ~150 | 加上 CHECKER.md (~153 行 ≈ 2200 token)，总计 ~2350 |
| REVISE | ~15 | ~200 | 加上 original_content 和 draft_summary，总计 ~800-1500 |

所有模板均在子代理或 O 的独立 context 中运行，不影响主 session 的 context budget。
