## 一、现状分析

### 1.1 四个工作流的编排状态

| 工作流 | 触发命令 | 编排方式 | MCP 工具 | SKILL.md 路由 | 状态 |
|--------|---------|---------|---------|-------------|------|
| **learn** | `/aristotle learn <query>` | ✅ MCP 编排（Function-Call-O） | `orchestrate_start` + `orchestrate_on_event` | 调 MCP → 执行 action | **已完成** |
| **reflect** | `/aristotle` (无参) | ⚠️ SKILL.md 直接加载 REFLECT.md | `create_reflection_record` / `complete_reflection_record` / `persist_draft`（辅助工具，非编排） | `Read REFLECT.md → 执行协议` | **半手动** |
| **review** | `/aristotle review N` | ⚠️ SKILL.md 直接加载 REVIEW.md | `read_rules` / `write_rule` / `reject_rule` 等（数据工具，非编排） | `Read REVIEW.md → 执行协议` | **半手动** |
| **sessions** | `/aristotle sessions` | ⚠️ SKILL.md 直接读 aristotle-state.json | 无 | `格式化显示 state file` | **硬编码** |

### 1.2 当前 reflect 流的执行路径

```
SKILL.md (40行) → 解析命令为 "reflect"
  → 直接加载 REFLECT.md (135行)
    → 主 session 执行 F1-F6 全部协议步骤
    → 主 session 需要: 解析参数、调 session_list、fire task()、
       调 MCP create_reflection_record、等 R 通知、fire C task()、
       调 MCP complete_reflection_record、输出通知
```

**问题：** 主 session 加载了 135 行协议文件，理解了 F1-F6 的全部语义，承担了 O 的决策职责。

### 1.3 当前 review 流的执行路径

```
SKILL.md (40行) → 解析命令为 "review"
  → 直接加载 REVIEW.md (180行)
    → 主 session 执行 V1-V6 全部协议步骤
    → 主 session 需要: 读 state file、读 DRAFT、调 MCP read_rules、
       格式化展示、处理用户 confirm/revise/reject、
       可选加载 CHECKER.md (153行) 执行验证、调 MCP write_rule/commit_rule
```

**问题：** 主 session 加载了 180-333 行协议文件，且 review 只能在发起 reflect 的同一 session 中执行。

### 1.4 当前 sessions 的执行路径

```
SKILL.md (40行) → 解析命令为 "sessions"
  → 直接读 aristotle-state.json
  → 格式化显示
```

**问题：** 逻辑硬编码在 SKILL.md 中，无 MCP 工具支撑。

### 1.5 learn 流的参考实现（已完成）

learn 是唯一通过 MCP 编排的工作流，其模式值得推广：

```
SKILL.md (~40行，不变)
  → 解析命令为 "learn"
  → 调用 MCP: orchestrate_start("learn", {query: "..."})
  → MCP 返回 {action: "fire_o", o_prompt: "...", workflow_id: "wf_xxx"}
  → SKILL.md 执行 fire_o action: task(prompt=o_prompt)
  → 等待 O 完成 → 调用 MCP: orchestrate_on_event("o_done", {...})
  → MCP 返回 {action: "notify", message: "🦉 Found N rules..."}
  → SKILL.md 展示通知 → DONE
```

**关键设计特征：**
1. SKILL.md 不知道任何协议语义，只执行 MCP 返回的 action
2. 所有决策在 MCP state machine 中完成
3. O subagent 按需调用（仅在需要 LLM 推理时）
4. 主 session context 保持干净

## 二、目标架构

### 2.1 统一 Function-Call-O 模式

将所有四个工作流纳入同一个编排模式：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: SKILL.md (~60 行)                              │
│                                                           │
│  ROUTE: 解析命令 → 调 MCP orchestrate_start → 执行 action  │
│  EVENT LOOP: notification → 调 MCP orchestrate_on_event   │
│  ACTIONS: fire_o / fire_sub / notify / done               │
│                                                           │
│  ⚠ 不加载任何协议文件（REFLECT.md / REVIEW.md / LEARN.md）│
│  ⚠ 不理解任何 GEAR 语义                                   │
│  ⚠ 不直接读写 state file                                  │
└───────────────────────┬──────────────────────────────────┘
                        │ MCP tool calls
                        ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 2: MCP Server — 状态机 + 纯逻辑决策                │
│                                                           │
│  orchestrate_start(command, args)                         │
│    "learn"    → 意图提取或直接搜索                         │
│    "reflect"  → 初始化状态，返回 fire_sub(R) action       │
│    "review"   → 加载 DRAFT + rules，返回 notify action    │
│    "review_action" → 处理用户 confirm/revise/reject       │
│    "sessions" → 格式化 state file，返回 notify action     │
│                                                           │
│  orchestrate_on_event(event_type, data)                   │
│    "subagent_done" → 根据 role 和 state 决定下一步         │
│    "o_done"        → 记录 O 的结果，决定下一步             │
│                                                           │
│  内部维护 workflow state (.workflows/{wf_id}.json)        │
│  所有"该做什么"的决策都在这里                              │
└───────────────────────┬──────────────────────────────────┘
                        │ fire subagent (主 session 执行)
                        ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Subagents — 按需调用的执行单元                   │
│                                                           │
│  R (Reflector) → REFLECTOR.md (独立 session)              │
│  C (Checker)   → CHECKER.md (独立 session)                │
│  O (LLM推理)   → MCP 提供的 prompt (无状态)               │
│  S (Scorer)    → MCP 提供的 prompt (无状态)               │
│                                                           │
│  协议文件只在 subagent context 中可见                       │
│  主 session 和用户永远看不到协议内容                        │
└─────────────────────────────────────────────────────────┘
```

### 2.2 MCP 工具扩展计划

#### 现有工具（14个）

```
State 辅助工具 (2):

```

#### 需要扩展的能力

| 工作流 | 需要的 MCP 能力 | 实现方式 |
|--------|---------------|---------|
| reflect | 初始化 workflow → 返回 fire_sub(R) → R 完成 → 返回 fire_sub(C) → C 完成 → 返回 notify | 扩展 `orchestrate_start` + `orchestrate_on_event` |
| review | 加载 DRAFT + rules → 格式化返回 | 新增 `orchestrate_review_load` 或复用 `orchestrate_start("review", ...)` |
| review (用户反馈) | 处理 confirm/revise/reject | 新增 `orchestrate_review_action(workflow_id, action, feedback)` |
| sessions | 格式化 state file | 复用 `orchestrate_start("sessions", ...)` |

**设计决策：** 尽量复用 `orchestrate_start` 和 `orchestrate_on_event`，新增 `orchestrate_review_action` 处理交互式 review。

### 2.3 统一 Action 协议

主 session 只需理解以下 action：

| Action | 主 session 执行 | 何时触发 |
|--------|---------------|---------|
| `fire_o` | `task(category="unspecified-low", run_in_background=true, prompt=o_prompt)` | learn 流需要 LLM 推理时 |
| `fire_sub` | `task(category="unspecified-low", load_skills=[], run_in_background=true, prompt=sub_prompt)` | reflect 流 fire R/C |
| `notify` | 展示 `message` 给用户 | 结果就绪 |
| `wait` | 不做任何事，等待下一个 notification | 中间状态 | *(reserved for future use)* |
| `done` | 流程结束 | 所有步骤完成 |

**注意：** `fire_sub` 是新增 action，区别于 `fire_o`：
- `fire_o`：fire O subagent（无状态 LLM 推理，不需要加载协议文件）
- `fire_sub`：fire R 或 C subagent（需要加载协议文件 REFLECTOR.md / CHECKER.md）

**说明：** `sub_role` 字段为信息性标注，与 workflow state 中的 `pending_role` 冗余。MCP 根据 `(command, phase)` 推断实际角色，SKILL.md dispatch 不依赖此字段。保留它是为了调试可读性。

### 2.4 GEAR 协议一致性

本方案满足 GEAR 2.0 协议的以下一致性要求：

| 一致性要求 | 本方案实现 | 映射 |
|-----------|---------|------|
| Function-Call-O 模式 | SKILL.md 为 O 的执行层，MCP 为 O 的决策层 | §2.1 三层架构 |
| 状态机驱动 | `orchestrate_start` + `orchestrate_on_event` 维护 phase 转换 | §3.1.1 状态机 |
| DRAFT → Review → Confirm | `persist_draft` → `orchestrate_review_action` → `commit_rule` | §3.2 |
| Δ 审核决策 | `get_audit_decision` 自动计算审核级别 | 现有 MCP 工具 |
| Git-backed 版本控制 | 所有规则通过 `commit_rule` 提交到 Git | 现有 MCP 工具 |
| Intent-driven 检索 | GEAR 2.0 frontmatter fields（domain, task_goal, failed_skill） | 现有 MCP 工具 |
| 跨 session 可用性 | DRAFT 从磁盘读取，不依赖 session_read | §3.2.4 |
| Checker 自动拉起 | R 完成 → `orchestrate_on_event` 自动 fire C | §3.1.3 |

#### 3.1.4 SKILL.md 路由变更

**Before（当前）：**
```
Otherwise: follow Parse Arguments section directly.
→ Read REFLECT.md → 执行 F1-F6
```

**After（目标）：**
```
Otherwise: call MCP orchestrate_start("reflect", {target_session_id, focus, ...}) → execute returned action.
```

**关键变化：** SKILL.md 不再加载 REFLECT.md。但 SKILL.md 需要在调 MCP 前解析 target_session_id（因为 MCP 无法调用 session_list）。

**SKILL.md 预解析逻辑（约 12 行）：**
```
## PRE-RESOLVE (reflect only)
If command is "reflect":
  1. Resolve target_session_id via session_list:
     - No argument → current session ID
     - "last" → session_list(limit=2), take first non-current
     - "session ses_xxx" → ses_xxx
     - "recent N" → session_list(limit=N+1), exclude current, take Nth
  2. If session_list fails or returns empty: notify user "🦉 Could not resolve target session. Use /aristotle session ses_xxx" and STOP
  3. Detect user_language from recent messages
  4. Get project_directory from cwd
  5. Call orchestrate_start("reflect", {target_session_id, focus, project_directory, user_language})
```

**为什么 SKILL.md 需要预解析：** MCP server 是纯被动服务，无法调用 `session_list()` 或获取当前 session 上下文。target 解析必须在主 session 中完成，然后将结果传给 MCP。

**PRE-RESOLVE 边缘情况处理：**
- `session_list` 调用失败（网络/权限）→ 提示用户使用 `/aristotle session ses_xxx` 直接指定
- `session_list` 返回空结果 → 提示 "🦉 No sessions found."
- `recent N` 中 N 超出范围 → 取最近可用 session，提示 "🦉 Only M sessions available, using most recent."
- 无法确定 current session ID → 降级为 `session_list(limit=1)` 取最新 session

### 3.2 Review 工作流

#### 3.2.1 状态机

Review 有两种模式：
1. **加载展示**（一次性）：加载 DRAFT + rules，格式化返回
2. **用户反馈处理**（交互式）：处理 confirm / revise / reject

```
orchestrate_start("review", {sequence: N})
         │
         ▼
  ┌──────────────┐     action: notify (DRAFT + rules 格式化内容)
  │ phase: review │ ────────────────────────────────────────► 主 session 展示
  └──────────────┘

用户: confirm / revise / reject / re_reflect
         │
         ▼
orchestrate_review_action(workflow_id, action, feedback)
         │
    ┌────┴────┬────────────┬──────────────┐
    │         │            │              │
  confirm   revise      reject       re_reflect
    │         │            │              │
    ▼         ▼            ▼              ▼
 commit    fire_o       reject_rule    fire_sub(R)
 staging  (LLM修改)     + notify       + 循环防护
 rules       │                         (max 3)
    │        ▼ (O 完成)                   │
    │  orchestrate_on_event               │
    │  ("o_done")                         ▼
    │        │                    新 sequence + parent
    ▼        ▼                    notify(new review N)
  notify  write_rule + commit
  (结果)    + notify
```

#### 3.2.2 orchestrate_start 扩展

```python
elif command == "review":
    sequence = args.get("sequence")
    if not sequence:
        return {"action": "notify", "message": "🦉 Usage: /aristotle review N"}
    
    # 加载 DRAFT + rules
    state_path = resolve_repo_dir().parent / "aristotle-state.json"
    records = json.loads(state_path.read_text(encoding="utf-8"))
    
    # 找到第 N 条记录
    target_record = None
    for i, r in enumerate(records):
        if i + 1 == sequence:
            target_record = r
            break
    
    if not target_record:
        return {"action": "notify", 
                "message": f"🦉 Reflection #{sequence} not found. Run /aristotle sessions to list."}
    
    # 读取 DRAFT 文件
    draft_path = target_record.get("draft_file_path", "")
    draft_content = ""
    if draft_path:
        dp = Path(draft_path).expanduser()
        if dp.exists():
            draft_content = dp.read_text(encoding="utf-8")
    
    # 读取相关规则
    # [CRITICAL-3 修复] 使用 list_rules 替代 read_rules 进行 keyword 搜索，
    # 避免加载完整规则内容（仅加载 metadata）。
    # Phase 2 应为 read_rules 添加 source_session 专用搜索维度。
    target_session = target_record.get("target_session_id", "")
    rules_result = list_rules(
        status_filter="all",
        keyword=target_session,
        limit=20,
```

### 4.5 首次使用自动初始化

方案一（推荐）：在 `orchestrate_start` 中增加自动 init 检测：

```python
def _ensure_repo_initialized():
    """确保 repo 已初始化，首次使用时自动初始化。"""
    repo_dir = resolve_repo_dir()
    if not (repo_dir / ".git").exists():
        init_repo()

def _cleanup_stale_workflows(max_age_hours: int = 24):
    """清理已完成或卡死的 workflow 文件。"""
    workflows_dir = resolve_repo_dir() / ".workflows"
    if not workflows_dir.exists():
        return
    cutoff = datetime.now(timezone.utc) - timedelta(hours=max_age_hours)
    stale_cutoff = datetime.now(timezone.utc) - timedelta(hours=max_age_hours * 2)
    for wf_file in workflows_dir.glob("*.json"):
        try:
            wf = json.loads(wf_file.read_text(encoding="utf-8"))
            phase = wf.get("phase", "")
            updated = wf.get("updated_at", "")
            if not updated:
                continue
            updated_dt = datetime.fromisoformat(updated)
            # 已完成的 workflow：超过 max_age_hours 清理
            if phase == "done" and updated_dt < cutoff:
                wf_file.unlink()
            # 卡死的 workflow（非终态超过 2x max_age_hours）：清理
            elif phase in ("reflecting", "checking", "review") and updated_dt < stale_cutoff:
                wf_file.unlink()
        except (json.JSONDecodeError, ValueError):
            pass
```

在 `orchestrate_start` 入口处调用 `_ensure_repo_initialized()` 和 `_cleanup_stale_workflows()`。

## 六、技术约束

### 6.1 硬约束

| 约束 | 值 | 理由 |
|------|---|------|
| SKILL.md 行数 | ≤60 行 | 控制主 session context 开销（含 ~10 行 PRE-RESOLVE 指令） |

| SKILL.md 不加载协议文件 | REFLECT.md / REVIEW.md / LEARN.md / CHECKER.md 不出现在 SKILL.md 中 | 架构隔离 |
| MCP 工具参数不暴露 | action 名和参数不在用户可见输出中出现 | 用户无需了解内部机制 |

| 现有测试不退化 | 134 pytest + 67 static 全部通过 | 回归保护 |

### 6.2 平台约束

| 约束 | 影响 | 应对 |
|------|------|------|

## 九、架构决策记录

### AD-1：SKILL.md 预解析 vs MCP 解析

**决策：** SKILL.md 预解析 target_session_id，结果传给 MCP。

**理由：**
- MCP 无法调用 `session_list()`（平台硬约束）
- MCP 无法获取当前 session ID
- 预解析逻辑简单（4 种 case），不会膨胀 SKILL.md

**替代方案：** 在 MCP 中硬编码 shell 命令解析 session_list 输出 → 违反 MCP 纯逻辑原则。

### AD-2：新增 fire_sub action vs 复用 fire_o

**决策：** 新增 `fire_sub` action，与 `fire_o` 分离。

**理由：**
- `fire_o` 用于无状态 LLM 推理（O subagent 不加载协议文件）
- `fire_sub` 用于有状态 subagent（R/C 加载协议文件）
- 两者参数不同：fire_o 只有 prompt，fire_sub 有 prompt + role
- 区分后 MCP 可以根据 role 做不同的状态转换

**补充说明（Round 1 Review）：** `sub_role` 为信息性字段，与 workflow state 中的 `pending_role` 冗余。MCP 实际根据 `(command, phase)` 推断角色进行 dispatch，SKILL.md 不依赖此字段做路由。保留用于调试可读性。

### AD-3：Review 交互逻辑位置

**决策：** Review 的用户交互逻辑（confirm/revise/reject）留在 SKILL.md，但决策逻辑在 MCP。

**理由：**
- SKILL.md 是唯一能与用户交互的地方
- MCP 处理数据操作（读 DRAFT、写规则、reject 规则）
- SKILL.md 只做"收到用户输入 → 调 MCP → 展示结果"的转发

### AD-4：保留 vs 删除 REFLECT.md

**决策：** Phase 1 保留，Phase 2 评估后决定。

**理由：**
- 保留作为回退路径
- 保留作为协议文档
- Phase 1 验证 MCP 编排稳定后再考虑删除
