## 1. 错误归因链

从发现"模型不遵循 SKILL.md"到找到真正根因，我经历了以下归因链：

| # | 归因 | 阶段 | 性质 | 被推翻原因 |
|---|------|------|------|-----------|
| 1 | "opencode run 不支持异步通知" | Layer 4 测试 | 正确但片面 | 只是测试方法问题，不是业务逻辑问题 |
| 2 | "模型不遵循 SKILL.md 指令" | 交互式测试 | 错误归因 | 模型遵循了 ROUTE，说明能读懂 SKILL.md |
| 3 | "LEARN.md 存在导致模型绕过 dispatcher" | 文件可见性分析 | 表面原因 | 删除文件能回避问题但不能解释原因 |
| 4 | "SKILL.md 的指令格式是文档风格而非执行风格" | 结构对比分析 | ✅ 根因 | 有对比证据（ROUTE vs ACTIONS 结构差异） |

每一步都是基于不完整信息做出的结论，然后在新证据出现后才修正。

## 2. 根因分析 (5-Why)

**Q1: 为什么做了 3 次错误归因？**
A: 每次观察到一个现象后，直接跳到最表面的解释，没有追问"为什么这个解释是正确的"。

**Q2: 为什么没有追问？**
A: 每个表面解释看起来都"合理"——opencode run 确实不支持异步通知，模型确实加载了 LEARN.md。这些事实是正确的，但结论是错误的。

**Q3: 为什么正确事实导致错误结论？**
A: 因为每个结论都只解释了部分现象，没有覆盖全部观察：
- "opencode run 问题"解释了 Layer 4 失败，但没解释交互式 session 中也失败
- "模型不遵循指令"解释了 learn 流程，但没解释 reflect 流程同样走 MCP
- "LEARN.md 存在"解释了为什么加载它，但没解释为什么不执行 ACTIONS

**Q4: 为什么没有检查部分解释是否覆盖全部现象？**
A: 没有在做结论前系统性验证——没有同时测试 reflect 流程来检查"模型不遵循指令"是否是全局问题。

**Q5: 根因？**
A: **单一观察点的解释偏好**——看到一个问题后，立即用最直接的因果关系解释它，而不检查这个解释是否覆盖所有相关现象。本质上是验证不充分。

## 3. 具体错误模式

### 错误模式 1：事实正确 ≠ 结论正确

| 事实 | 正确结论 | 我做的错误结论 |
|------|---------|--------------|
| opencode run 不支持异步通知 | 测试方法需要调整 | "整个方案在 opencode 上不可行" |
| 模型加载了 LEARN.md | ACTIONS 部分指令不够清晰 | "模型不遵循指令" |
| LEARN.md 文件可见 | 文件存在是一个诱因 | "删除文件就能解决" |

### 错误模式 2：没有做反向验证

在声称"reflect 遵循而 learn 不遵循"时：
- ❌ 没有测试 reflect 流程验证这个声称
- ❌ 没有检查两个流程的结构差异
- ✅ 用户质问后才做了实测，发现 reflect 也不遵循 SKILL.md 的文件加载指令

### 错误模式 3：跳到方案而不是理解问题

从"模型不遵循指令"直接跳到"删文件"方案：
- ❌ 没有先问"为什么 ROUTE 被遵循但 ACTIONS 不被遵循"
- ❌ 没有先对比成功 skill 的指令格式
- ✅ 用户质问"为什么方案是删文件"后才回到问题分析

## 4. 教训

### 教训 1：做结论前做反面验证

当声称"A 行为而 B 不行为"时，必须验证 B 的行为。如果两个都行为异常，根因可能在更深的层面。

**自检问题**：我的结论是否覆盖了所有已观察到的现象？有没有未覆盖的异常？

### 教训 2：先理解问题再给方案

"删除文件"是一个方案，不是一个分析。当用户问"为什么"时，应该先给出因果链，再提出方案。用户质问"为什么方案是删文件"正是这个错误。

**自检问题**：我是否理解了"为什么"再回答"怎么做"？

### 教训 3：对比分析优于单点分析

当某个组件行为异常时，找到一个"正常"的同类组件做对比，差异就是线索。ROUTE 和 ACTIONS 在同一个文件里，直接对比就发现了格式差异。

**自检问题**：异常组件和正常组件之间的结构性差异是什么？

### 教训 4：表面归因需要覆盖全部观察

每个结论发布前检查：这个解释是否覆盖了所有已知的异常行为？如果只覆盖了部分，说明还有更深的原因。

**自检问题**：列出所有异常行为，逐条检查结论是否解释了它。

## 5. 预防规则

1. **做结论前列出全部异常行为**，确认结论覆盖每一条
2. **声称 X 行为而 Y 不行为时**，必须实测 Y 的行为
3. **方案提出前**，先完成因果链分析（why chain ≥ 3 层）
4. **事实正确但结论可能错误**——每个事实可以有多个解释，选覆盖最广的那个

## 6. 第三次错误归因（2026-04-21 补充）

### 错误模式：行为观察 ≠ 机制确认

| # | 观察 | 错误结论 | 真实情况 |
|---|------|---------|---------|
| 1 | `opencode run` 测试结果不一致 | "opencode run 不支持异步通知" | 未验证，可能是其他原因 |
| 2 | `opencode run` 回调链走通 | "opencode run 支持异步通知" | 未验证，可能是同步 fallback |

两次错误是**同一模式的正反面**：从表面行为推断底层机制，都没有做区分性实验。

### 关键发现

回调链走通可能由两种完全不同的机制产生：
- **机制 A**：异步通知（`<system-reminder>` 投递到 session）
- **机制 B**：同步等待（task tool 在 run mode 下降级为同步执行）

两种机制产生相同的输出，但只有机制 A 是产品的真实运行路径。

### 预防规则补充

5. **行为观察 ≠ 机制确认。** 看到回调链走通不等于确认了异步通知机制。确认机制需要区分性实验——寻找只有目标机制才会产生的证据（如 `<system-reminder>` 标记、执行时间线差异、源码路径追踪）。

**自检问题**：我观察到的行为，能否用我声称的机制之外的其他机制解释？如果能，需要做什么实验来排除？
