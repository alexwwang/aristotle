## 1. 测试目标

验证 SKILL.md dispatcher 在真实 opencode session 中的行为：

| 验证项 | 验证内容 | 对应设计要求 |
|--------|---------|-------------|
| V1 | explicit params → 直接 MCP search → notify（跳过 O） | 函数调用确定性 |
| V2 | NL query → orchestrate_start → fire_o action | MCP 正确路由 |
| V3 | fire_o → task() → O subagent 后台执行 | 模型执行 task() 而非加载 LEARN.md |
| V4 | O 完成 → orchestrate_on_event → list_rules → notify | 异步回调链 |
| V5 | workflow state 文件正确创建和更新 | 状态持久化 |
| V6 | 主 session 不泄露协议内部术语 | 上下文隔离 |
| V7 | reflect 命令 → 加载 REFLECT.md（非 MCP 路径） | dispatcher 路由隔离 |

## 2. 测试路径

根据反思文档结论，Layer 4 分为两条路径：

### 路径 A：同步路径（`opencode run`）

| 场景 | 命令 | 预期行为 | 验证项 |
|------|------|---------|--------|
| S2 | `/aristotle learn --domain database_operations --goal connection_pool` | orchestrate_start → list_rules → notify | V1, V6 |

`opencode run` 对同步路径可行：模型调用 MCP orchestrate_start，MCP 检测到明确参数直接走 search，返回 notify action，模型显示结果后 STOP。无异步回调。

### 路径 B：异步路径（tmux 交互式 session）

| 场景 | 命令 | 预期行为 | 验证项 |
|------|------|---------|--------|
| S1 | `/aristotle learn 数据库连接池超时怎么处理` | orchestrate_start → fire_o → task() → 通知 → orchestrate_on_event → list_rules → notify | V2, V3, V4, V5 |
| S3 | 在 S1 的 session 中问"回顾刚才的执行过程" | 无协议内部术语泄露 | V6 |
| S5 | `/aristotle` | 加载 REFLECT.md 并执行 reflect 协议 | V7 |

**为什么必须用 tmux**：
- `opencode run` 是单次命令模式（源码确认：`packages/opencode/src/cli/cmd/run.ts`）
- 后台 task 完成后投递 `<system-reminder>` 通知到交互式 session
- fire_o 流程依赖这个通知触发 orchestrate_on_event 回调
- 单次模式下无通知机制，fire_o 的 step 2 永远不会触发

## 4. 测试执行步骤

### 4.1 路径 A（同步，opencode run）

```bash
# S2: explicit params
opencode run --model <MODEL> \
  --title "Layer4-S2-Learn-Explicit" \
  "/aristotle learn --domain database_operations --goal connection_pool"
```

**断言**（3 条）：
- [ ] 输出包含 🦉 或搜索结果
- [ ] 输出不包含协议内部术语（GEAR, intent_extraction, LEARN.md）
- [ ] 完成时间 < RUN_TIMEOUT（默认 120s，无 O subagent）

### 4.2 路径 B（异步，tmux 交互式）

```bash
# 创建 tmux session
tmux new-session -d -s layer4-test -x 200 -y 50

# 启动 opencode 交互式 session
tmux send-keys -t layer4-test "opencode --model <MODEL>" Enter

# 等待 session 就绪（提示符出现）
sleep 10

# S1: NL query → fire_o 流程
tmux send-keys -t layer4-test "/aristotle learn 数据库连接池超时怎么处理" Enter

# 等待 O subagent 完成和回调（可能需要 30-120s）
# 监控完成标志：🦉 前缀的输出出现
sleep 60

# 捕获 session 输出
tmux capture-pane -t layer4-test -p -S -500 > /tmp/layer4-s1-output.txt
```

**S1 断言**（5 条）：
- [ ] S1-P1: 流程在 timeout 内完成
- [ ] S1-P2: 模型执行了 `task()` 启动 O subagent（正向验证）且未加载 LEARN.md（反向验证）— **V3 核心**
- [ ] S1-P3: 最终输出包含 🦉 + 搜索结果
- [ ] S1-P4: 无协议内部术语泄露
- [ ] S1-P5: 流程到达终态（未卡在 fire_o 循环）

**S1 的核心验证**是 V3：模型是否执行了 `task()` 而非加载 LEARN.md。这是 SKILL.md 格式修复的直接验证。

**V3 验证方法（正负双维度）**：
- **正向**：输出中包含 `task(` / `background task` / `subagent` / `spawning` → 证明模型执行了 SKILL.md 的 fire_o action
- **反向**：输出中**没有** `Reading/Loading LEARN.md` → 证明模型没有走替代路径
- 判定逻辑：正向 + 反向 → PASS；仅反向 → WARN（不确定）；反向失败 → FAIL
- 可通过 session_read 查看详细 tool calls 确认

```bash
# S3: context cleanliness（在同一 session）
tmux send-keys -t layer4-test "请回顾刚才 /aristotle learn 的完整执行过程" Enter
sleep 15
tmux capture-pane -t layer4-test -p -S -200 > /tmp/layer4-s3-output.txt
```

**S3 断言**（2 条）：
- [ ] S3-P1: 回顾中无协议内部术语（intent_extraction, phase_search, list_rules, frontmatter, yaml）
- [ ] S3-P2: 回顾中包含 skill 介入执行的证据（MCP / subagent / orchestrate / skill / 后台 / 通知 关键词）— 验证 SKILL.md 确实被加载执行

```bash
# S5: reflect 路由（路由隔离验证）
tmux send-keys -t layer4-test "/aristotle" Enter
sleep 10
tmux capture-pane -t layer4-test -p -S -200 > /tmp/layer4-s5-output.txt
```

**S5 断言**（2 条）：
- [ ] S5-P1: reflect 协议特有标记出现（`Reflector subagent` / `STEP F3` / `STEP R[1-5]` / `DRAFT report` / `session_read` / `aristotle-state.json`）— 证明 REFLECT.md 被加载
- [ ] S5-P2: reflect 未调用 `orchestrate_start`（路由隔离正确）

### 5.2 Path B — S1（异步路径，交互式 session）

**步骤 1：启动交互式 session**

```bash
# 新开终端窗口，启动 opencode
cd /Users/alex/aristotle-coroutine-o
opencode --model <MODEL>
```

等待 session 完全加载（出现输入提示符）。

**步骤 2：执行 S1 命令**

在 opencode TUI 中输入：
```
/aristotle learn 数据库连接池超时怎么处理
```

**步骤 3：等待完成**

观察输出直到出现 🦉 前缀的最终消息。可能需要 30-120 秒（O-agent 后台执行 + 回调）。

**步骤 4：记录输出**

复制 TUI 中**所有可见内容**（包括 Thinking 行、⚙ 工具调用行、正文输出），保存到文本文件。

**断言判定**：

| 断言 | 判定方法 | PASS 条件 |
|------|---------|----------|
| S1-P1: 超时内完成 | 检查从输入到 🦉 出现的时间 | < 120s |
| S1-P2: V3 核心（正向） | 检查输出（含 Thinking） | 出现 `task(` 或 `background task` 或 `subagent` |
| S1-P2: V3 核心（反向） | 检查输出（含 Thinking） | **没有** `Reading/Loading LEARN.md` |
| S1-P3: 搜索结果 | 检查最终 🦉 消息 | 含搜索结果或 "No relevant lessons" |
| S1-P4: 无协议泄漏 | **排除 `⚙│└▣` 开头的行后**，检查剩余文本 | 无 GEAR/Reflector/intent_tags/intent_extraction/5-Why/CRITICAL ARCHITECTURE/orchestrate_start/orchestrate_on_event/fire_o/o_prompt |
| S1-P5: 终态 | 检查最终输出 | 含 🦉 或 "done" 或 "complete" 或 "没有找到" |

**S1-P4 判定细节**：

```
# 需要排除的行（TUI 自动渲染，非模型主动输出）：
⚙ aristotle_orchestrate_start [command=learn, ...]       ← 排除
⚙ aristotle_orchestrate_on_event [data_json={...}]       ← 排除
│ General Task — ...                                      ← 排除
└ 0 toolcalls · 13.8s                                    ← 排除
▣ Sisyphus - Ultraworker · ...                           ← 排除

# 需要检查的行（模型主动输出）：
O agent 已启动，正在分析学习意图，等待结果返回...          ← 检查
🦉 No relevant lessons found for this query.              ← 检查
Aristotle 的规则库中目前没有...                            ← 检查
```

| 断言 | 判定方法 | PASS 条件 |

1. 模型是否加载了 REFLECT.md（Thinking 中应出现 "Read REFLECT.md" 或类似表述）
2. 是否执行了 reflect 协议的步骤（session_list → task() Reflector → create_reflection_record）

**步骤 3：记录输出**

复制**所有可见内容**（含 Thinking）。

**断言判定**：

| 断言 | 判定方法 | PASS 条件 |
|------|---------|----------|

### 5.7 测试顺序和依赖

```
Path A (S2)          ← 独立，可随时执行
    │
Path B (同一 session):
    ├─ S1 (fire_o)   ← 第一个，建立 session 上下文
    ├─ S3 (context)  ← 依赖 S1 的 session 上下文
    └─ S5 (reflect)  ← 依赖 S1/S3 的 session 上下文
    │
S4 (filesystem)      ← 在所有交互完成后检查
```

> Path B 的 S1 → S3 → S5 **必须在同一 opencode session 中顺序执行**，因为 S3 需要回顾 S1 的执行过程，S5 测试 reflect 路由隔离。

## 6. 通过标准

| 条件 | 要求 |
|------|------|
| 总断言数 | 15 条 |
| 通过率 | ≥93%（14/15），S3-P1 为已知局限（见下） |
| 核心验证 V3 | 必须通过（模型执行 task() 而非加载 LEARN.md） |
| 核心验证 V7 | 必须通过（reflect 走 REFLECT.md 而非 MCP） |

### S3-P1 已知局限

S3-P1（模型回顾暴露协议术语）降级为"建议通过"。

**理由**：S3 的测试场景是用户主动要求回顾内部执行过程（"请回顾刚才的执行过程"）。V6（无协议泄漏）的核心风险是模型**被动泄漏**协议术语 — 在正常使用中未经用户询问就暴露内部机制。S1/S2 中模型主动输出已经 PASS，说明被动泄漏已被有效防止。S3 场景下模型暴露内部术语是对用户直接提问的如实回答，不属于 V6 要防范的风险。

## 7. 失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| S1 V3 失败：模型仍加载 LEARN.md | ACTIONS 格式仍不清晰，需要进一步调研指令格式 |
| S1 V3 失败：模型忽略 SKILL.md 完全 | 检查 skill 安装路径和 SKILL.md 可见性 |
| S2 失败 | MCP search 链路问题，检查 Layer 1-3 测试 |
| S5 失败 | reflect 路由逻辑问题，检查 Parse Arguments 部分 |
| 全部超时 | 模型或 provider 问题，换模型重试 |
