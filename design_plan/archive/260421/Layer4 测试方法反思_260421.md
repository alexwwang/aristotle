# Layer 4 测试方法错误 — 反思文档

**日期**: 2026-04-21
**分支**: coroutine-O
**相关规则**: rec_1776768630 (PATTERN_VIOLATION)

---

## 1. 错误摘要

Layer 4 live test 使用 `opencode run --format json`（非交互式单次命令）验证 Function-Call-O 架构的异步通知回调机制。被测架构要求交互式 session 环境（后台任务通知 → 主 session 回调），但测试框架不支持此环境。导致：

- 3 次测试结果不一致（12/15 → 13/15 → 12/15），每次结果不同
- 在错误的测试框架上做了 5 轮修补（改断言、改 timeout、改 SKILL.md、改 extract_texts），每轮"改善"给人收敛错觉
- 最终归因为"非确定性 LLM 行为"，实则是测试方法根本不匹配

## 2. 错误链

```
第1步: 直接复用 live-test.sh 的 opencode run 模式
       ↓ 未评估适用性
第2步: 测试结果不一致（S4 有时通过有时失败）
       ↓ 错误归因
第3步: 归因为"LLM 非确定性行为"而非"测试方法不匹配"
       ↓ 在错误层面修复
第4步: 修改 SKILL.md、test.sh、timeout、extract_texts
       ↓ 每次修改后重测，结果渐进改善
第5步: "在收敛"的错觉 → 声称"MVP 验证通过"
       ↓
结论: 虚假验证
```

## 3. 根因分析 (5-Why)

**Q1: 为什么 Layer 4 测试结果不一致？**
A: `opencode run` 单次模式下，后台任务完成后不投递通知。模型有时走直接路径（内联处理语义推理），有时走异步路径（fire_o 等通知），两种路径产生不同结果。

**Q2: 为什么选了 `opencode run`？**
A: 现有 `live-test.sh` 用此模式，直接复用了。

**Q3: 为什么复用前没有评估？**
A: 没有先分析被测架构的运行环境需求，就直接选了测试工具。

**Q4: 为什么没分析运行环境？**
A: 设计方案指定了 tmux 交互式测试，实现时为了简化改用非交互模式，但没有评估这个变更的影响。

**Q5: 根因？**
A: **实现偏离设计时没有做适配性评估。** 从 tmux 方案改为 `opencode run` 是一个正确的简化决策（降低复杂度），但没有检查这个简化是否丢弃了被测逻辑赖以运行的必要条件（异步通知机制）。

## 4. 教训

1. **先确认环境，再选工具。** 测试框架的选择是设计决策，不是实现细节。选择前必须明确被测程序的真实运行环境。

2. **渐进修补是危险信号。** 如果测试需要多轮小修补才能通过，停下来检查是否在错误的轨道上。

3. **不一致的结果 ≠ 非确定性行为。** 不一致的测试结果首先归因于测试方法的缺陷，只有排除了测试方法问题后才归因于被测逻辑。

4. **偏离设计时要做 impact check。** 简化实现可以，但每个简化都要检查：是否丢弃了被测功能的核心前置条件？

## 5. 修正方案

Layer 4 测试应分为两条路径：

| 路径 | 测试环境 | 验证内容 | 测试工具 |
|------|---------|---------|---------|
| 同步路径 | `opencode run` | explicit params → orchestrate_start → list_rules → notify | 现有脚本，仅 Scene 2 |
| 异步路径 | 交互式 session | NL query → fire_o → O subagent → 通知 → orchestrate_on_event → list_rules → notify | tmux + opencode 交互式 |

异步路径的测试需要：
1. tmux 创建交互式 opencode session
2. 发送 `/aristotle learn <query>` 命令
3. 等待 O subagent 完成通知
4. 验证主 session 的回调行为
5. 检查 workflow state 文件

这与原设计方案的 Layer 4 测试方案一致 — 回到设计文档的 tmux 方案。

## 6. 补充记录（2026-04-21 第二次测试）

### 实测发现

`opencode run` 模式下 fire_o 回调链可以完整执行（orchestrate_start → task() → background_output → orchestrate_on_event → list_rules → notify），但这**不改变本文档的核心结论**。

### ⚠️ 第三次错误归因：从"不支持"到"支持"

在发现 `opencode run` 走通了完整回调链后，我立即下了一个新结论："opencode run 支持异步通知，之前的结论是错的"。这和第一次错误归因是**同一模式的反面**：

| 次序 | 观察 | 错误结论 | 验证？ |
|------|------|---------|--------|
| 第一次 | 测试结果不一致 | "opencode run 不支持异步通知" | ❌ 未验证 |
| 第二次 | 回调链走通 | "opencode run 支持异步通知" | ❌ 未验证 |

两次都是从表面行为推断底层机制，都没有做区分性实验。

### 回调链走通的两种可能机制

**机制 A（异步通知）**：
1. `task(run_in_background=true)` → 后台 task 异步执行
2. 后台 task 完成 → `<system-reminder>` 通知投递到 session
3. 模型看到通知 → 调用 `background_output` 获取结果 → 调用 `orchestrate_on_event`

**机制 B（同步等待/fallback）**：
1. `task(run_in_background=true)` → 但 `opencode run` 单次模式下改为同步等待
2. task 完成后直接返回结果（无通知投递）
3. 模型收到结果 → 调用 `orchestrate_on_event`

两种机制产生**相同的输出**，但本质完全不同：
- 机制 A 是产品真实路径（交互式 session 中用户实际经历的流程）
- 机制 B 是 fallback 行为（单次模式下的退化路径）

**当前状态：** 未做任何区分性实验来确定实际是哪种机制。待调研 opencode task tool 的实际执行方式。

### 为什么 `opencode run` 仍然不是正确的测试方式

被测程序（Aristotle skill）的真实运行环境是**交互式多轮 session**：
1. 用户在交互式 session 中输入 `/aristotle learn <query>`
2. 后台 O subagent 完成后，通知投递到**同一个交互式 session**
3. 模型在同一 session 的上下文中继续处理后续指令

`opencode run` 是**单次命令模式**——每次执行是独立进程、独立上下文。即使它能完成单次 fire_o 回调链，但：
- 没有模拟"同一 session 内多轮交互"的真实场景
- S3（context cleanliness）需要在同一 session 回顾之前的执行——`opencode run` 无法测试
- S5（reflect 路由隔离）需要在前序 learn 执行之后的 session 中触发——`opencode run` 无法模拟
- 模型在单次模式下的行为可能与交互式模式下不同（上下文、系统提示、skill 加载方式等）
- **即使回调链走通，也无法确认走的是异步通知路径还是同步 fallback 路径**

**结论不变：测试方式必须匹配被测程序的真实运行环境。** 交互式场景必须用交互式测试。tmux send-keys 与 opencode TUI 不兼容是待解决的**测试基础设施问题**。

### 教训补充

**教训 5：行为观察 ≠ 机制确认**

看到回调链走通不等于确认了异步通知机制。相同的输出可以由不同的底层机制产生。确认机制需要**区分性实验**（discriminating experiment），而非表面观察。

区分性实验的例子：
- 在输出中搜索 `<system-reminder>` 通知标记（异步路径的证据）
- 对比 `opencode run` 和交互式 session 中 task 执行的时间线差异
- 检查 opencode 源码中 task tool 在 run mode 下的行为定义
- 在 `opencode run` 中设置断点或日志确认通知投递

**自检问题**：我观察到的行为，能否用我声称的机制之外的其他机制解释？如果能，需要做什么实验来排除？
