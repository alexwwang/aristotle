# P3.1 设计讨论：学习链路中 O 的角色定位

> 日期：2026-04-16
> 类型：设计决策记录

---

## 背景

P3.1（LEARN.md）需要确定学习检索链路的交互模式。两种候选方案：

**方案 A：L 直连 S**
```
L → S（函数调用）→ MCP read_rules → 结果 → L
```
L 直接调用 S 函数构造查询、执行 MCP 检索、接收原始结果。

**方案 B：O 居间协调**
```
L → O（Aristotle skill）→ S → MCP → O 过滤压缩 → L
```
L 向 O 发出学习请求，O 提取意图、委托 S 检索、过滤压缩结果后返回精炼摘要给 L。

---

## 第一次评估（错误）

Agent 选择了方案 A（L 直连 S），理由：

1. **O 在学习链路中是不必要的中间人。** 学习是读操作，无副作用，不需要协调者审批。
2. **O 声称做的三件事都没必要：** "判断是否需要检索"——L 自己最清楚；"过滤结果"——read_rules 已做过 status 过滤；"压缩摘要"——可以做成 S 输出格式的一部分。
3. **"在 Aristotle 的实际实现中，O 和 L 是同一个 agent（主会话）。L → O → S → O → L 本质上是同一个 agent 穿过四个概念角色跟自己对话。"**
4. 引用 CQRS 模式做类比：命令走协调者，查询直接取。

---

## 用户指出错误

> "我认为你有一个关键误解：L 是当下和用户交互的那个 agent，它在做用户的一个主线任务，而 O 是给 L 提供反思能力的主 agent，这两个 agent 并不是同一个。Aristotle 的目的是为了将反思放在主线任务之外，尽可能减小对主线任务上下文的污染，这是核心原则。"

核心纠正：**L 和 O 不是同一个 agent。** L 是执行用户主线任务的 agent（比如正在写代码），O 是 Aristotle 这个独立 skill。它们运行在不同的上下文中。

---

## 错误根因分析（5-Why）

**Why 1：为什么认为 O 和 L 是同一个 agent？**

设计文档的角色映射表中，O 映射到 `SKILL.md`（主会话加载），L 也映射到"OpenCode 主会话中的 agent"。两边都写着"主会话"，直接划了等号。

**Why 2：为什么看到"主会话"就划等号？**

Aristotle 的现有实现（P1+P2）确实是同一个 agent 进程在不同时刻扮演多个角色——加载 SKILL.md 变成 O，加载 REFLECT.md 启动反思，加载 REVIEW.md 做审核。全部在同一个主会话上下文中完成。L 的运行场景（P3）从未实现过，所以用已有的 O 的运行模式去推断了 L。

**Why 3：为什么用 O 的模式去推断 L？**

混淆了"agent 进程"和"上下文状态"。在 LLM agent 系统中，同一个 agent 进程在不同时刻可以有完全不同的上下文——加载了 SKILL.md 时的上下文，和没加载 SKILL.md 正在帮用户写代码时的上下文，是两个不同的东西。没有在 context window 层面建模，而是在进程层面建模。

**Why 4：为什么没有在 context window 层面建模？**

"最小化上下文污染"这个原则在脑中是一个文档级的设计目标，而不是一个架构约束。把它当作"好的实践"而不是"不可违反的硬约束"。所以在权衡"简洁性（L 直连 S）"和"隔离性（O 居间过滤）"时，优先了简洁性。

**Why 5：根本原因**

对"间接层"有默认的负面判断——认为每多一层协调就是不必要的复杂度。这个判断在一般软件设计中通常是合理的，但在 Aristotle 的场景下是错误的。Aristotle 的间接层不是开销，它是**产品本身**。整个 skill 的存在意义就是让反思基础设施对主线 agent 不可见。去掉间接层等于去掉产品价值。

---

## 修正后的评估：方案 B（O 居间协调）

O 必须在学习链路中居间，原因如下：

### 1. L 不能加载 LEARN.md

LEARN.md 是 Aristotle 的内部协议文件。加载到 L 的上下文里就违背了最小污染原则。L 只需要发出一个轻量请求，接收一个精炼摘要。

### 2. S 的查询构造和 MCP 调用不能发生在 L 的上下文里

调用 `read_rules`、解析 frontmatter、遍历结果——这些都是反思基础设施的细节。L 的上下文应该留给用户的主线任务。

### 3. 过滤和压缩必须在 O 侧完成

MCP 可能返回 20 条规则，L 不应该看到原始结果。O 提取每条规则的 `error_summary` + `Proposed Rule` + `Example`，压缩成几行摘要，只把这个给 L。

### 修正后的交互链路

```
L（主线 agent，不知道 GEAR 的存在）
  │
  │ "我要做数据库迁移"（自然语言，不感知反思系统）
  │
  ▼
O（Aristotle，独立 skill 上下文）
  │
  │ 从 L 的自然语言中提取 intent_tags
  │ 调用 S → 构造 MCP 查询 → 获取原始结果
  │ 过滤 + 压缩为精炼摘要
  │
  ▼
L 收到 3-5 条精炼的教训摘要（不感知来源机制）
```

### L 的视角

L 只知道自己问了一个问题，收到了几条教训。不知道 MCP、不知道 `read_rules`、不知道 frontmatter。这是最小污染。

### 与 REFLECT/REVIEW 的一致性

这和现有的反思/审核模式一致——主会话 agent 不加载 REFLECTOR.md，只收到 DRAFT 摘要。学习链路遵循同样的隔离原则。

### 间接层的定位

O 居间不是"增加了复杂度"，而是**架构必需**。O 是 L 和反思基础设施之间的隔层。L 不需要知道反思系统怎么运作，只需要 O 返回的结论。

---

## 决策结论

**采用方案 B。** LEARN.md 定义 O 居间的学习协议，L 不直接接触反思基础设施。

### LEARN.md 的四个部分

1. **O 的意图提取**：从 L 的自然语言请求中提取 `intent_tags`（L 不需要知道什么是 intent_tags）
2. **O → S → MCP**：查询构造、执行、获取原始结果（全部在 O 的上下文中）
3. **O 的过滤与压缩**：去重、排序、提取关键信息
4. **O → L 的输出格式**：精炼摘要（只有 error_summary + 避坑要点 + 正反例）

### O 在学习链路中仅有的两个协调点

- **入口**：被动触发（P3.3）——O 从对话中检测到错误信号，主动提醒 L 去学习
- **出口**：错误反馈升级——L 学习后仍犯错，向 O 报告，O 拉起 R/C 开始新一轮反思

---

## 提取的预防规则

> **上下文隔离是架构约束，不是设计偏好。** 当设计文档声明"最小化 X 对 Y 的污染"时，任何在 X 和 Y 之间去除隔层的提议都必须被否决，除非同时证明了隔层去除后污染不会增加。不能仅凭"更简洁"就拆除隔离层。
