## 1. 设计原则

**核心约束：上下文隔离。** L 是执行用户主线任务的 agent，O 是 Aristotle 独立 skill。L 的上下文必须尽可能留给用户的主线任务，反思基础设施的任何细节（MCP、frontmatter、查询构造）都不应进入 L 的上下文。

**读写分离的分界：**
- **写路径**（R → C）：O 协调。写操作有副作用，需要质量门禁。
- **读路径**（L → O → S → O → L）：O 居间。读操作无副作用，但 O 作为隔离层保护 L 的上下文。

**与现有架构的一致性：** REFLECT/REVIEW 模式中，主会话 agent 不加载 REFLECTOR.md，只收到 DRAFT 摘要。学习链路遵循同样的隔离原则——L 不加载 LEARN.md，只收到精炼摘要。

## 2. 完整交互链路

```
L（主线 agent）
  │
  │ ① 自然语言请求（不知道 GEAR 的存在）
  │   例："我要做数据库迁移，之前踩过坑吗？"
  │   或 O 主动触发（P3.3 被动触发场景）
  │
  ▼
O（LEARN.md 中扮演知识服务协调者）
  │
  │ ② 意图提取：从 L 的自然语言中推断 intent_tags
  │    domain = "database_operations"
  │    task_goal = "schema_migration"
  │
  │ ③ 委托 S（O 内的函数调用）
  │
  ▼
S（函数调用，非独立 subagent）
  │
  │ ④ 查询构造：intent_tags → read_rules 参数
  │ ⑤ 执行 MCP 调用：read_rules(intent_domain="database", ...)
  │ ⑥ 返回原始结果集（可能 0-20 条）
  │
  ▼
O（过滤器角色）
  │
  │ ⑦ 相关性过滤：去掉 domain 匹配但 task_goal 无关的规则
  │ ⑧ 数量控制：如果命中 >5 条，只保留最相关的 5 条
  │ ⑨ 摘要压缩：每条规则提取 error_summary + 避坑要点 + 正反例
  │
  ▼
L（收到精炼摘要）
  │
  │ ⑩ 解读与应用：调整执行策略，规避已知错误模式
  │
  │ ⑪ 如果学习后仍犯错 → 向 O 提交错误现场报告
```

## 3. LEARN.md 文档结构

LEARN.md 分为四个部分，全部在 O 的上下文中执行，L 不可见。

### Part A：O 的意图提取

**触发场景：**

| 场景 | 触发方式 | 说明 |
|------|---------|------|
| 主动触发 | 用户执行 `/aristotle learn` | L 主动请求学习 |
| 参数化触发 | 用户执行 `/aristotle learn --domain X --goal Y` | 显式指定检索维度 |
| 被动触发 | O 从对话中检测到错误信号（P3.3） | 自动拉起学习流程 |

**意图提取规则：**

O 从 L 的自然语言中推断以下信息：

```
学习请求（O 内部结构，L 不可见）:
  intent_tags:
    domain: "推断的技术领域"
    task_goal: "推断的任务目标"
  failed_skill: "上一次失败的 skill/tool（可选，L 不一定提供）"
  error_context: "简要错误描述（可选，用于 error_summary 匹配）"
```

**domain 推断参考表（与 REFLECTOR.md 一致）：**

| 关键词信号 | domain |
|-----------|--------|
| 文件读写、路径、权限 | `file_operations` |
| API 调用、HTTP、外部服务 | `api_integration` |
| 数据库、查询、ORM、迁移 | `database_operations` |
| 语法错误、代码生成 | `code_generation` |
| 编译、打包、依赖 | `build_system` |
| 测试、断言 | `testing` |
| CI/CD、部署、配置 | `deployment` |
| 无法明确归类 | `general` |

**阈值评估：**

- `intent_tags.domain` 非空 → 可以检索
- `intent_tags.domain` 为空但 `error_context` 非空 → 用 `keyword` 参数做全文匹配
- 两者都为空 → O 告知 L 信息不足，请求补充或终止

### Part B：S 的查询构造逻辑

S 是 O 内的一个函数调用，输入是 O 提取的学习请求，输出是 `read_rules()` 的参数字典。

**构造规则：**

```
params = {status: "verified"}  // L 只看已验证的规则

if intent_tags.domain 非空:
  params.intent_domain = intent_tags.domain

if intent_tags.task_goal 非空:
  params.intent_task_goal = intent_tags.task_goal

if failed_skill 非空:
  params.failed_skill = failed_skill

if error_context 非空:
  // 从 error_context 中提取 2-3 个关键词用 | 连接
  params.keyword = extract_keywords(error_context)
```

所有参数 AND 组合 → `read_rules(**params)`

**关键词提取策略（error_context → keyword）：**

取 error_context 中的核心技术名词，用 `|` 连接用于正则匹配：
- 例："Prisma connection pool timeout" → keyword = "prisma|timeout|pool"
- 例："TypeScript circular import" → keyword = "circular|import"

### Part C：O 的过滤与压缩

**相关性过滤：**

S 返回原始结果后，O 做二次过滤：
1. **domain 精确匹配**优先：task_goal 完全匹配的排前面
2. **去重**：同一 category 下如果有多条相似规则，保留 error_summary 最具体的
3. **数量控制**：最多返回 5 条。超过 5 条时按 relevance 排序截断

**摘要压缩格式：**

每条规则压缩为以下格式（不返回完整规则正文）：

```
⚠ [category] error_summary（一句话描述错误场景）
  避坑：Proposed Rule 的核心约束（1-2 句）
  ✅ 正确做法 | ❌ 错误做法
  规则ID: rec_xxx（供错误反馈时引用）
```

**示例输出（L 收到的内容）：**

```
🦉 Found 3 relevant lessons from past sessions:

1. ⚠ [HALLUCINATION] Prisma connection pool exhaustion in serverless
   避坑：Serverless 环境下必须显式设置 connection_limit=1，默认值 10 会导致连接池耗尽
   ✅ DATABASE_URL + "?connection_limit=1" | ❌ 使用默认 Prisma 连接配置
   规则ID: rec_1713283200

2. ⚠ [PATTERN_VIOLATION] Migration 文件必须包含 down 方法
   避坑：每个 migration 文件必须同时实现 up() 和 down()，否则回滚会失败
   ✅ 总是成对编写 up/down | ❌ 只写 up 忽略 down
   规则ID: rec_1713283456

3. ⚠ [SYNTAX_API_ERROR] Prisma $executeRaw 不支持模板字符串插值
   避坑：$executeRaw 中必须用 Prisma.sql 标签或参数化查询，不能直接拼接
   ✅ Prisma.sql`SELECT * FROM ${table}` | ❌ `SELECT * FROM ${table}`
   规则ID: rec_1713283789
```

### Part D：L 的结果应用与错误反馈

**L 的应用策略：**

L 收到精炼摘要后：
1. **error_summary** 作为关键警告——在执行当前任务时主动规避
2. **避坑要点** 作为约束——在生成代码/方案时内联遵循
3. **正反例** 作为对照——参照正确做法
4. **规则ID** 记录下来——如果学习后仍犯错，反馈时引用

**错误反馈流程：**

L 学习后仍犯错时，向 O 提交"错误现场报告"：

```
错误现场报告:
  intent_tags: {domain: "...", task_goal: "..."}
  failed_skill: "..." (如适用)
  applied_rules: ["rec_xxx", "rec_yyy"]  ← 学习过但未能避免的规则 ID
  error_description: "..."
```

O 收到错误报告后的处理：
1. 标记对应规则为 `needs_sync`
2. 拉起 R → 新一轮反思（传入错误现场报告作为上下文）
3. R 生成改进方案（新 pending 文件）
4. C 审核 → verified / rejected

## 4. L 的接口边界

**L 知道的：**
- 可以向 Aristotle 请求学习（自然语言）
- 收到精炼的教训摘要（error_summary + 避坑 + 正反例 + ID）
- 犯错后可以向 Aristotle 反馈

**L 不知道的（O 隔离）：**
- GEAR 协议的存在
- MCP、`read_rules`、frontmatter 等基础设施
- intent_tags 的具体字段名和格式
- 查询构造逻辑
- 规则文件在磁盘上的存储位置和格式

| 文件 | 角色 | LEARN.md 中的交互 |

## 7. 设计决策记录

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| O 是否居间 | 是（方案 B） | L 直连 S（方案 A） | 上下文隔离是架构约束，O 是 L 和反思基础设施的隔层 |
| S 的定位 | O 内函数调用 | 独立 subagent | 已在 GEAR 2.0 方案中确认，先轻量实现 |
| 结果格式 | 精炼摘要 | 完整规则正文 | 最小化 L 上下文污染 |
| 数量限制 | 最多 5 条 | 不限制 | 避免信息过载 |
| domain 推断 | O 从自然语言推断 | L 显式提供 | L 不应知道 domain 字段的存在 |
