好的，这是你要的形式化可行性分析报告：

** coexistence 可行，核心在于「 session 创建仍走 OpenCode 原生 API，但执行流完全绕过阻塞式 `prompt()`，改用 `promptAsync()` + SSE 事件驱动」，从而在 OMO-Slim 的阻塞式 `task` 世界之外，开辟一条并行的、非侵入式的异步执行通道。**

**推荐架构为：独立部署的「异步任务调度插件」（建议命名为 `standalone-async-bridge` 或类似），注册一套以 `async_` 为前缀的独立工具（如 `async_delegate`、`async_retrieve`），内部基于 `promptAsync` 创建会话、通过监听 `session.idle` 等事件感知任务结束，并将结果持久化到本地磁盘，实现与主 session 的非阻塞交互。该方案与 OMO-Slim 的共存方式不是「修改 OMO-Slim 的行为」，而是「在 OMO-Slim 的盲区建立独立生态」，具备物理隔离级别的抗干扰能力。**

# OpenCode 插件异步化与主会话非阻塞交互可行性研究报告

## 1. 核心问题定义与现状诊断

### 1.1 问题域界定

在现代 AI 辅助编程工具链中，OpenCode 作为一个可扩展的 AI 代码助手平台，其插件架构允许开发者通过自定义插件来增强核心功能。然而，随着插件生态的复杂化，特别是当多个大型插件（如 OMO 和 OMO-Slim）共存时，会话（Session）管理和异步执行机制成为制约用户体验的关键瓶颈。本报告聚焦于一个具体的技术挑战：如何在 OpenCode 平台上实现一个能够异步与主会话（Main Session）交互的插件，且该插件的执行不受其他主要插件（特别是 OMO-Slim）阻塞行为的影响。当前的情况是，一个插件在 OMO 环境下可以依赖其回调机制正常运行，但在 OMO-Slim 环境下，由于 OMO-Slim 的主会话会被后台运行的子代理（Subagent）阻塞，导致无法实现用户无感知的任务执行效果。因此，本研究的目标是探索并实现一种更可控的异步执行机制，确保插件能够与主会话共存且互不干扰，最终达成真正的“后台运行、前台无忧”的用户体验。

该问题的核心在于 OpenCode 原生 `task` 工具的同步阻塞特性。当主会话（Main Session）通过 `task` 工具委派任务给子代理（Subagent）时，它会创建一个子会话（Child Session）并调用阻塞式的 `Session.prompt()` 方法。这意味着主会话必须等待子代理完全执行结束后才能继续处理用户的其他指令。这种设计在某些简单场景下是可行的，但在复杂的插件生态中，尤其是当 OMO-Slim 这样的编排插件频繁使用子代理进行后台处理时，主会话的交互线程会被长时间占用，导致终端用户界面（TUI）假死或无法响应输入。用户所期望的“无感”效果，即任务在后台执行的同时，主会话保持完全可交互状态，在这种同步模型下是无法实现的。因此，问题的关键在于如何绕过或替代这种原生的阻塞式调用，建立一条并行的、非阻塞的通信与执行通道。

### 1.2 OMO 与 OMO-Slim 的架构差异分析

OMO (Oh My OpenCode) 和 OMO-Slim 是两个在 OpenCode 生态中广泛使用的插件，它们都提供了多代理编排的能力，但在实现复杂度和对原生机制的依赖上存在显著差异。理解这些差异是诊断阻塞问题的根源并提出有效解决方案的前提。OMO 作为一个功能全面的超级插件，其核心哲学是“最小化认知负担”和“可预测、可委托” [^4^]。它构建了一套完整的代理生态系统，包括 11 个以上的专业代理（如 Sisyphus、Prometheus、Hephaestus 等），并通过复杂的钩子（Hook）系统和工具注册机制与 OpenCode 进行深度集成 [^19^][^20^]。OMO 的内部实现可能包含了对原生 `task` 工具的封装或通过其自定义的 `delegate_task` 工具来管理子代理的调用。关键在于，OMO 的完整版可能通过其内部的回调和事件处理机制，在一定程度上缓解了原生阻塞调用带来的问题，或者其设计模式使得阻塞发生在用户不直接感知的环节。

相比之下，OMO-Slim 是 OMO 的一个精简版（slimmed-down fork），其核心聚焦于代理编排（Agent Orchestration），剥离了许多额外的功能 [^13^]。它同样采用“中心辐射”（Hub and Spoke）模型，由一个编排器（Orchestrator）将任务分发给专门的代理（Spokes），并包含一个后台管理器（Background Manager）来处理“即发即弃”（Fire-and-forget）的任务 [^13^]。尽管 OMO-Slim 的设计理念支持后台任务，但其实现方式可能更直接地依赖于 OpenCode 的原生 API。社区中存在的已知 Bug 报告揭示了问题的本质：在使用 OMO 插件时，无论主代理是哪个，一旦存在多个子会话且其中一个仍在流式输出，OpenCode 的 TUI 就会间歇性地停止接受交互 [^11^]。这个 Bug 在纯 OpenCode 核心环境中无法复现，只有在启用 OMO 插件时才会出现，这表明 OMO 或其衍生版本（如 OMO-Slim）在管理多个子会话时，其内部机制与 OpenCode 的 UI 渲染或输入处理循环发生了冲突。OMO-Slim 可能继承了这一问题，或者由于其更轻量级的实现，对原生阻塞调用的副作用更为敏感，导致其主会话更容易被后台运行的子代理阻塞。

| 特性 | OMO (Oh My OpenCode) | OMO-Slim |
| :--- | :--- | :--- |
| **核心定位** | 功能全面的超级插件，提供完整的 AI 编码工作流 | OMO 的精简版，聚焦于核心的多代理编排功能 [^13^] |
| **代理数量** | 11+ 个专业代理（Sisyphus, Prometheus 等） | 代理数量较少，核心为编排器和专门代理 [^13^] |
| **架构模型** | Hub-and-Spoke 多代理协作 | Hub-and-Spoke 模型，包含后台任务管理器 [^13^] |
| **功能特性** | 包含 Ultrawork、Hash-Anchored Edits、LSP 等高级功能 [^4^] | 精简功能，主要保留代理委派和后台任务能力 [^13^] |
| **与原生 API 关系** | 深度集成并扩展原生 API，可能包含自定义阻塞管理逻辑 | 更直接依赖原生 API，可能暴露了原生 `task` 工具的阻塞问题 |
| **已知问题** | 存在子会话流式输出时导致 TUI 无响应的 Bug [^11^] | 用户反馈主会话易被后台子代理阻塞，无法实现无感执行 |

### 1.3 阻塞问题的技术根因

#### 1.3.1 原生 `task` 工具的同步执行模型

OpenCode 平台提供了一个原生的 `task` 工具，旨在允许代理将一个复杂的任务委派给另一个专门的子代理（Subagent）来处理。然而，该工具在设计上采用了同步阻塞的执行模型，这是导致主会话被卡住的技术根因。通过对 OpenCode 内部机制的深入分析，可以发现 `task` 工具的实现逻辑如下：当一个代理（父代理）决定使用 `task` 工具时，它首先通过 OpenCode SDK 的 `Session.create()` 方法创建一个新的子会话（Child Session）。这个子会话在会话树中被设置为当前主会话的子节点，继承了一部分上下文，但拥有独立的执行环境。紧接着，`task` 工具会调用 `Session.prompt()` 方法，将需要执行的任务指令发送给新创建的子会话中的子代理 [^10^]。

问题的关键就在于 `Session.prompt()` 这个调用。这是一个同步（或表现为同步的 `await`）操作，父代理的当前执行线程会在此处暂停，直到子代理处理完任务并返回结果。这意味着，从用户的角度看，一旦一个 `task` 调用被发起，主会话就必须等待这个后台任务完全结束后才能继续。这种设计在一些简单的、顺序依赖的任务场景中是合理的，但它完全不适用于需要并行处理或后台运行的复杂工作流。例如，如果用户希望主代理在后台进行代码搜索或文档分析的同时，自己继续在前台编写代码或与主代理进行其他交互，这种同步模型就会成为一个巨大的障碍。主会话的交互循环被阻塞，导致用户界面无法响应新的输入，直到后台的 `task` 调用返回。

#### 1.3.2 `Session.prompt()` 的阻塞特性对主会话的影响

`Session.prompt()` 方法的阻塞特性对主会话（Main Session）的交互体验产生了直接且严重的影响。在 OpenCode 的 TUI（终端用户界面）中，主会话负责处理用户的输入、与主代理（Primary Agent）进行交互，并渲染代理的输出。当主代理通过 `task` 工具触发一个子代理任务时，由于 `Session.prompt()` 的同步等待，主代理的执行流程被挂起。这种挂起状态会级联到整个主会话的交互循环中。具体表现为，在子代理执行任务期间，TUI 可能无法接受新的用户输入，或者即使能够输入，也无法得到主代理的及时响应。用户会感觉到界面“冻结”或“卡顿”，尤其是在子代理执行耗时较长的任务（如深度代码分析、网络请求等）时，这种体验会非常糟糕。

更为严重的是，这种阻塞行为还会影响到其他插件的运行。如果一个插件（例如用户想要开发的插件）依赖于与主会话的异步交互，那么当 OMO-Slim 或其他插件通过 `task` 工具启动了阻塞性的子代理任务时，该插件的异步任务同样会被卡住，因为它们共享同一个主会话的事件循环或执行上下文。GitHub 上关于 Oh My OpenCode 的一个 Bug 报告精确地描述了这一问题：当存在多个子会话，并且其中一个正在流式输出内容时，用户将无法在父会话和子会话之间进行切换，也无法在输入框中输入内容 [^11^]。这证实了 `Session.prompt()` 的阻塞特性不仅仅是逻辑上的等待，它还会对 OpenCode 的 UI 渲染和事件处理造成实际干扰，使得实现真正的“用户无感”后台任务变得不可能。因此，任何旨在实现异步、非阻塞交互的插件方案，都必须从根本上规避对原生 `task` 工具及其同步 `Session.prompt()` 调用的依赖。

## 2. OpenCode 会话与插件机制深度剖析

### 2.1 OpenCode 插件架构与 Hook 系统

#### 2.1.1 插件入口与生命周期

OpenCode 的插件系统是一个基于 TypeScript 的、运行在主进程内的扩展框架，它允许开发者通过一系列明确定义的钩子（Hooks）来拦截、修改或增强 OpenCode 的核心行为。与一些使用外部进程或标准输入/输出流进行通信的代码助手（如 Claude Code 的 hooks 机制）不同，OpenCode 的插件直接在主进程中运行，这使得它们能够与 OpenCode 的内部状态和 API 进行更深度的、更低延迟的交互 [^5^]。一个插件的入口通常是一个导出的异步函数，该函数接收一个包含上下文的 `PluginInput` 对象，并返回一个实现了 `Plugin` 接口的对象。这个返回的对象包含了插件想要注册的各种钩子处理函数。在 OpenCode 启动时，它会扫描并加载配置的插件，调用每个插件的入口函数，并将注册的钩子整合到核心的事件处理流程中。

插件的生命周期与 OpenCode 主进程紧密相连。当 OpenCode 初始化时，插件被加载，其入口函数被执行，完成钩子的注册。在运行期间，当特定的事件发生时（例如，一个工具即将被执行，或一个会话被创建），OpenCode 会遍历所有注册了对应钩子的插件，并按顺序调用它们的处理函数。这种事件驱动模型是插件系统的核心。例如，一个插件可以注册 `tool.execute.before` 钩子来在所有工具执行前进行权限检查，或者注册 `event` 钩子来监听会话的生命周期事件，如 `session.created` 或 `session.idle` [^2^][^35^]。这种机制为构建复杂的扩展功能（如 OMO 的多代理系统或 Daytona 的沙盒集成）提供了坚实的基础。然而，也正因为插件运行在同一个进程内，一个插件的错误（如未捕获的异常或向 stderr 写入数据）可能会直接影响到 OpenCode 的稳定性，甚至破坏 TUI 的显示 [^34^]。

#### 2.1.2 核心 Hook 点：`tool`, `event`, `tool.execute.before`

OpenCode 的插件 API 提供了一系列功能强大的 Hook 点，使得插件能够介入到核心工作流的各个关键环节。对于实现异步、非阻塞的插件交互而言，以下几个 Hook 点尤为重要：`tool`、`event` 和 `tool.execute.before`。`tool` Hook 允许插件向 OpenCode 的代理注册全新的自定义工具。这是一个非常强大的功能，因为它使得插件可以为 AI 代理提供原本不具备的能力。例如，一个插件可以注册一个名为 `async_delegate` 的工具，该工具在内部实现了非阻塞的任务委派逻辑。当 AI 代理在决策过程中认为需要执行一个可以异步处理的任务时，它就可以调用这个由插件提供的 `async_delegate` 工具，而不是调用原生的、阻塞式的 `task` 工具。通过这种方式，插件可以“引导”AI 使用异步的交互模式，从而从根本上避免主会话被阻塞。

`event` Hook 则提供了一个监听 OpenCode 内部事件的机制。这些事件涵盖了会话的整个生命周期，包括 `session.created`（会话创建）、`session.idle`（会话空闲）、`session.deleted`（会话删除）、`message.updated`（消息更新）等 [^36^]。对于异步任务管理来说，`event` Hook 是实现“回调”或“通知”功能的关键。例如，当一个通过 `promptAsync` 启动的后台任务完成时，其对应的会话状态会变为 `idle`，并触发一个 `session.idle` 事件。插件可以通过监听这个事件来得知任务已经完成，然后执行后续操作，比如将结果持久化到磁盘，或者通过 `session.prompt({noReply: true})` 向父会话发送一个完成通知 [^2^][^35^]。值得注意的是，目前 OpenCode 在处理插件的 `event` 钩子时，是“即发即弃”（fire-and-forget）的模式，即不会等待插件事件处理函数返回的 Promise 完成 [^32^]。这意味着插件的异步事件处理不会阻塞 OpenCode 的核心事件流，但也要求插件自身要妥善处理异步逻辑，避免未捕获的异常。

`tool.execute.before` 和 `tool.execute.after` 这两个 Hook 点允许插件在工具执行前后介入。`tool.execute.before` 在处理函数的输出中提供了一个 `proceed` 布尔值，如果将其设置为 `false`，则可以阻止该工具的执行 [^17^]。这个功能可以用来实现权限控制或根据特定条件拦截工具调用。虽然对于构建新的异步流程不是直接必需的，但它可以用于增强异步任务的安全性。例如，可以拦截对原生 `task` 工具的调用，并根据策略决定是否允许其执行，或者将其重定向到自定义的异步实现。综合这些 Hook 点，插件架构提供了构建一个完全独立的、非阻塞的异步任务系统的所有必要原语，而无需修改 OpenCode 的核心代码。

### 2.2 OpenCode SDK 核心 API 分析

#### 2.2.1 `Session.create()`：会话树结构

`Session.create()` 是 OpenCode SDK 中用于创建新会话的核心 API，它在构建任何类型的多会话交互（无论是同步还是异步）中都扮演着基础性的角色。该方法允许创建独立的会话，或者通过在参数中指定 `parentID` 来创建一个会话树中的子会话。例如，`client.session.create({ body: { title: "Delegation: task-123", parentID: "session-id-of-parent" } })` 这样的调用会创建一个新的子会话，该会话在逻辑上隶属于父会话 [^29^]。这种会话树结构是 OpenCode 管理复杂交互和上下文继承的基础。在原生 `task` 工具的实现中，子会话就是通过这种方式创建的，这使得用户可以通过 TUI 中的快捷键（如 `Ctrl+X` 组合键）在父会话和子会话之间进行导航，形成一个清晰的层级关系 [^1^]。

对于实现异步插件而言，`Session.create()` 同样是第一步。当插件的 `async_delegate` 工具被调用时，它需要首先创建一个新的会话来运行后台任务。这个会话可以是一个带有 `parentID` 的子会话，这样可以保持与主会话的逻辑关联，便于后续的结果通知和会话管理。尽管 `opencode-background-agents` 插件声称其后台任务运行在“会话树之外”，但其实现代码中仍然使用了 `parentID` 来创建会话 [^29^]。这表明，至少在当前的 OpenCode SDK 中，完全脱离会话树的会话创建可能不是一个官方支持的模式，或者保持父子关系有利于管理和清理。重要的是，仅仅是创建一个子会话并不会导致阻塞。阻塞问题发生在下一步，即如何与该会话进行交互。因此，`Session.create()` 本身是一个安全、同步且快速的操作，它为后续的异步执行提供了容器。

#### 2.2.2 `Session.prompt()`：同步阻塞调用

`Session.prompt()` 是 OpenCode SDK 中用于向一个特定会话发送提示（prompt）并获取其响应的核心 API，同时也是导致主会话阻塞的“罪魁祸首”。这是一个同步 API，意味着调用方（通常是主会话中的代理）会在这个方法调用处暂停执行，并一直等待，直到目标会话中的代理处理完该提示并生成一个完整的响应后，方法才会返回结果。在原生 `task` 工具的实现中，一旦通过 `Session.create()` 创建了子会话，紧接着就会调用 `Session.prompt()` 来触发子代理执行任务。这个调用会阻塞主代理的当前执行线程，直到子代理任务完成，从而形成了“主会话等待子会话”的同步依赖关系 [^10^]。

这种同步阻塞的特性在需要即时反馈的交互中是必要的，但在后台任务场景下却成为了严重的性能瓶颈。如果一个任务需要执行几分钟甚至更长时间，主会话在这整个期间都将处于不可用状态。一个 GitHub 上的功能请求明确指出了这个问题，并提出需要一个“fire-and-forget”的机制，让主代理在委派任务后立即恢复交互状态，同时在子代理完成时通过某种信号或回调来接收通知 [^6^]。`Session.prompt()` 的设计显然不支持这种模式。因此，任何旨在实现异步后台任务的插件都必须完全避免在自己的核心逻辑中 `await` 或同步调用 `Session.prompt()`。虽然它在插件内部用于一些辅助操作（如 `opencode-background-agents` 在生成元数据时会创建一个临时会话并 `await` `session.prompt()` [^29^]），但对于启动后台任务这一关键路径，必须使用其他非阻塞的替代方案。

#### 2.2.3 `Session.promptAsync()`：非阻塞调用与 204 响应

`Session.promptAsync()` 是 OpenCode SDK 中为实现真正的异步、非阻塞执行而提供的关键 API。与同步阻塞的 `Session.prompt()` 不同，`promptAsync` 在调用后会立即返回一个 HTTP 204 No Content 响应，而无需等待目标会话中的代理完成处理 [^31^][^38^]。这意味着调用方（插件或主会话）可以在启动一个后台任务后立即恢复执行，继续处理其他事件或用户输入，从而实现了“fire-and-forget”的效果。这个 API 是实现用户无感知的后台任务执行的核心基石。当插件接收到一个异步任务委派请求时，它可以调用 `client.session.promptAsync({ path: { id: newSessionId }, body: { agent: "subagent-name", parts: [...] } })` 来启动任务。由于这个调用是非阻塞的，它不会占用主会话的执行线程。

`promptAsync` 的真正威力在于它与 OpenCode 事件系统的结合。因为调用方不会直接收到任务的执行结果，所以它需要通过监听事件来追踪任务的进度和完成状态。当一个通过 `promptAsync` 启动的后台任务最终完成时，其对应的会话会变为 `idle` 状态，或者会产生其他相关的事件（如 `message.updated`）。插件可以通过注册 `event` Hook 来捕获这些事件，并在事件处理函数中执行相应的逻辑，例如从会话中提取最终结果、将结果持久化到磁盘，并向父会话发送一个通知。一个 GitHub 上的功能请求提案明确指出了这一点：`promptAsync` + SSE (Server-Sent Events) 已经实现了后台执行的核心功能， missing piece 只是用于编排的“胶水”代码 [^31^]。`promptAsync` 消除了阻塞，事件系统提供了完成通知机制，两者结合为构建一个健壮的异步插件提供了完整且高效的解决方案。

#### 2.2.4 SSE (Server-Sent Events) 事件流机制

Server-Sent Events (SSE) 是 OpenCode 平台提供的一种服务器向客户端推送实时更新的机制，对于实现基于 `Session.promptAsync()` 的异步任务监控至关重要。OpenCode 内部维护着一个事件总线（Event Bus），各种内部状态变更（如会话创建、消息更新、会话空闲等）都会被发布为这个事件流中的事件。插件可以通过特定的 API 端点（如 `GET /event` [^31^]）订阅这个事件流，从而实时地获取到 OpenCode 内部发生的各种变化。这种基于推送的模式比轮询（polling）要高效得多，因为它只在有事件发生时才会发送数据，大大减少了不必要的网络请求和资源消耗。

在异步插件的架构中，SSE 扮演着“神经系统”的角色。当插件通过 `promptAsync` 启动一个后台任务后，它需要知道这个任务何时完成、是否成功、以及产生了什么结果。通过订阅 SSE 事件流，插件可以监听与后台任务会话 ID 相关的事件。最关键的事件之一是 `session.idle`。当一个后台任务会话处理完所有待办事项并进入空闲状态时，OpenCode 会发布一个 `session.idle` 事件，其中包含了该会话的 ID。插件的事件处理函数在接收到这个事件后，就可以确定对应的异步任务已经完成。随后，插件可以执行后续操作，例如调用 `client.session.messages()` API 来获取该会话中的所有消息，从而提取出子代理的最终输出。然后，插件可以将这个结果保存到文件中，并使用 `session.prompt({noReply: true})` 向父会话注入一个系统消息，告知用户或主代理任务已完成，结果已准备好 [^22^]。这种事件驱动的架构使得整个异步流程既高效又松耦合。

### 2.3 现有解决方案的启示

#### 2.3.1 `opencode-background-agents` 插件的实现机制

`opencode-background-agents` 是一个广受欢迎的开源插件，其目标正是为 OpenCode 带来类似 Claude Code 的后台代理（background agents）功能。它通过提供一套新的工具来实现异步委派，其核心机制为我们设计新的插件提供了宝贵的实践经验。该插件主要提供了三个工具：`delegate`（启动后台任务）、`delegation_read`（获取任务结果）和 `delegation_list`（列出所有任务）[^1^][^22^]。其工作流程是：用户或主代理调用 `delegate` 工具，提供任务描述和要使用的子代理类型。插件在接收到这个工具调用后，会执行一个非阻塞的启动流程。首先，它创建一个新的子会话。然后，它调用 `client.session.prompt()` 来启动子代理，但**关键之处在于，它并没有 `await` 这个 `prompt()` 调用** [^29^]。相反，它将这个 Promise 附加了一个 `.catch()` 处理函数来处理错误，然后立即返回，将控制权交还给主会话。

为了感知任务的完成情况，`opencode-background-agents` 插件深度集成了 OpenCode 的 `event` Hook。它注册了一个事件处理器，专门监听 `session.idle` 事件。当一个后台任务会话完成工作并进入空闲状态时，插件的事件处理器会被触发。在该处理器中，插件会执行一系列收尾工作：首先，它会调用 `client.session.messages({ path: { id: backgroundSessionId } })` 来获取后台会话的完整消息历史，从而提取出子代理的最终响应。接着，它会将这些结果格式化并持久化到本地磁盘上的一个特定目录（`~/.local/share/opencode/delegations/`）[^1^]。最后，它会通过向父会话发送一个带有特殊标记（如 `<task-notification>`）的系统消息来通知主代理任务已完成，并附上结果文件的路径。这种机制实现了任务的持久化和上下文的解耦，即使主会话经历了上下文压缩（compaction），它仍然可以通过 `delegation_read` 工具从磁盘读取之前后台任务的结果，因为这些结果已经存在于会话历史之外。

#### 2.3.2 其采用 `session.prompt()` (非 await) + `session.idle` 事件的模式

`opencode-background-agents` 插件的核心技巧在于对 `session.prompt()` 的非阻塞式使用。虽然 `session.prompt()` 本身是一个返回 Promise 的异步操作，但在 JavaScript/TypeScript 中，如果你不显式地使用 `await` 关键字或 `.then()` 来等待其完成，代码的执行会立即继续到下一行。`opencode-background-agents` 正是利用了这一语言特性。在其 `delegate` 方法的核心执行逻辑中，我们可以看到如下代码模式：

```javascript
// Fire the prompt (using prompt() instead of promptAsync() to properly initialize agent loop)
this.client.session.prompt({...})
  .catch((error: Error) => {
    void this.finalizeDelegation(delegation.id, "error", error.message)
  })
return delegation
```

这里，`this.client.session.prompt(...)` 返回的 Promise 没有被 `await`。代码立即执行到 `return delegation`，将包含任务 ID 的委托记录对象返回给调用者（主代理）。与此同时，在后台，那个未被等待的 `prompt()` 调用继续执行，驱动着子代理在另一个会话中处理任务。当这个后台的 `prompt()` 最终完成（无论成功或失败）时，附加在其上的 `.catch()` 处理器会被触发，用于处理错误情况。对于成功完成的情况，插件则完全依赖于 `session.idle` 事件。这种“fire-and-forget”的模式，结合事件驱动的状态追踪，巧妙地绕过了 `session.prompt()` 的阻塞特性，实现了主会话与后台任务的并行执行。

然而，这种模式也存在一个隐含的缺点。由于 `prompt()` 没有被 `await`，如果它在执行过程中抛出了一个未被 `.catch()` 捕获的异常，可能会成为一个未处理的 Promise rejection。尽管在现代 Node.js 版本中这通常不会导致进程崩溃，但它可能会产生一些警告或不可预测的行为。更重要的是，这种模式的可靠性在很大程度上取决于 `session.idle` 事件的准确性和及时性。如果由于某种原因（例如，子代理陷入了无限循环或 OpenCode 内部错误），`session.idle` 事件没有被正确触发，那么这个后台任务就可能会被插件“遗忘”，一直处于一种未知的运行状态，尽管实际上它可能已经停止或失败。

#### 2.3.3 该方案的局限性：仅限只读子代理

`opencode-background-agents` 插件虽然在实现异步执行上提供了一个可行的范例，但它自身带有一个重大的局限性，这直接影响了其适用范围。根据官方文档和 README 的明确说明，该插件的 `delegate` 工具**只能用于“只读”（read-only）的子代理** [^1^][^22^]。这里的“只读”是指子代理的权限被严格限制，不能对文件系统进行写操作。具体来说，这些子代理的配置必须满足 `edit=deny`、`write=deny` 和 `bash={"*": "deny"}` 等权限限制。如果子代理拥有任何写权限（例如，可以修改代码文件或执行 bash 命令），那么 `opencode-background-agents` 插件会明确拒绝执行委派，并提示用户应该使用原生的 `task` 工具。

这个限制的根本原因在于后台任务与 OpenCode 的撤销/分支（undo/branching）系统之间的不兼容性。OpenCode 维护了一个精细的撤销历史，允许用户回退到之前的状态。这个历史是与会话树紧密关联的。当用户执行一个操作时，这个操作会被记录在撤销历史中。当一个后台任务在会话树之外（或非阻塞地）运行时，它对项目文件所做的任何修改都将**无法被 OpenCode 的撤销系统追踪**。如果用户随后在主会话中执行了撤销操作，主会话的状态会回退，但后台任务所做的修改仍然会保留在文件系统中，这就会导致数据不一致和意外的数据丢失 [^1^]。例如，一个后台任务修改了文件 A，然后用户在主会话中执行了一个编辑操作并决定撤销。撤销操作会回退主会话的编辑，但不会回退后台任务对文件 A 的修改，从而可能导致代码逻辑错误。因此，`opencode-background-agents` 的开发者为了规避这种破坏性的副作用，选择了最安全的策略：只允许对文件系统无副作用的只读操作在后台执行。这个局限性意味着，对于需要进行代码生成、重构或任何文件修改的“写”操作，该插件方案并不适用，用户必须回归到原生的、阻塞式的 `task` 工具。

## 3. 异步插件共存方案设计

### 3.1 设计目标与约束条件

在设计一个能够与 OMO-Slim 等主要插件共存的异步插件时，我们必须首先明确核心设计目标和需要满足的约束条件。这些目标和约束共同构成了我们技术选型和架构设计的决策框架。首要目标是实现**真正的非阻塞交互**，即插件启动的后台任务绝不能以任何形式阻塞主会话（Main Session）的交互循环。无论后台任务执行多长时间，主会话的 TUI 都必须保持对用户输入的响应，主代理也必须能够继续处理新的指令。这是解决用户当前痛点的根本要求。其次是**与 OMO-Slim 等主要插件的共存性**。我们的插件不能依赖于修改 OMO-Slim 的源码或行为，也不能对其内部工作机制作出任何假设。共存应该是通过 OpenCode 提供的标准插件 API 实现的，确保在不同插件组合下都能稳定工作。

**独立性和隔离性**是另一个关键目标。插件的异步执行机制应该是一个独立的、自包含的模块，不依赖于其他插件的回调或状态。它应该有自己的任务调度、状态追踪和结果通知机制，避免与其他插件产生耦合或冲突。最后是**可控性与可观测性**。用户或主代理应该能够方便地启动、监控和取消后台任务。任务的状态（运行中、已完成、失败）应该是可查询的，执行结果也应该能够被可靠地获取。我们需要避免后台任务变成“失控的”或“被遗忘的”进程。这些目标共同指向了一个基于 OpenCode 原生 API（特别是 `promptAsync` 和事件系统）的、高度独立的插件设计。

### 3.2 方案一：基于 `promptAsync` + SSE 的独立异步调度插件

#### 3.2.1 核心思想：创建独立会话，通过事件驱动完成任务闭环

本方案的核心思想是构建一个完全独立的插件，该插件通过 OpenCode SDK 提供的 `Session.promptAsync()` API 和 Server-Sent Events (SSE) 事件流，建立一套与原生同步 `task` 工具并行的、非阻塞的异步任务调度系统。这个插件将注册一套全新的、以 `async_` 为前缀的工具（例如 `async_delegate`、`async_retrieve`、`async_list`），供主代理调用。当主代理需要执行一个可以异步处理的任务时，它会调用 `async_delegate` 工具，而不是阻塞式的 `task` 工具。插件在后台接收到这个工具调用后，会负责整个任务的调度、执行监控和结果通知的闭环，而整个过程对主会话是完全非阻塞的。

这个方案的本质是在 OMO-Slim 等插件所依赖的阻塞式世界之外，开辟一条全新的、基于事件驱动的异步通信通道。它不试图修改或拦截 OMO-Slim 的行为，而是提供一种更优的替代方案。通过 `promptAsync` 启动后台任务，主会话的执行线程得以立即释放。插件自身则转型为一个轻量级的任务调度器，它通过监听 SSE 事件来被动地感知任务状态的变化，而不是主动地、阻塞地等待任务完成。这种事件驱动的架构不仅实现了非阻塞的目标，还带来了良好的松耦合性和可扩展性。任务的执行者（后台会话中的子代理）和任务的调度者（插件）之间通过事件总线进行间接通信，彼此无需知道对方的存在，从而实现了高度的隔离。

#### 3.2.2 实现流程

##### 3.2.2.1 插件注册 `async_delegate` 工具

实现的第一步是在插件中注册一个新的工具 `async_delegate`。这个工具将作为主代理启动异步任务的入口。在插件的 `tool` Hook 中，我们需要定义这个工具的名称、描述和参数模式（schema），以便 OpenCode 的代理能够理解和使用它。`async_delegate` 的参数应该至少包含 `prompt`（任务描述）和 `agent`（要使用的子代理名称），还可以包括可选参数如 `timeout`（超时时间）。当主代理决定调用 `async_delegate` 时，它会传递这些参数，OpenCode 的插件系统会路由这个工具调用到我们插件的 `execute` 函数。这个 `execute` 函数是整个异步流程的起点，它的设计和实现必须遵循非阻塞的原则，即它应该尽快地启动后台任务并立即返回一个初步的结果（例如，一个任务 ID），而不是等待任务执行完毕。

##### 3.2.2.2 工具内部调用 `Session.create()` 和 `Session.promptAsync()`

在 `async_delegate` 工具的 `execute` 函数内部，实现流程分为两个关键步骤。首先，调用 `client.session.create()` 创建一个新的会话。这个会话将作为后台任务的执行容器。我们可以为其设置一个有意义的 `title`，并可以指定 `parentID` 为当前主会话的 ID，以便在会话树中建立逻辑关联，但这并非强制要求。创建会话后，我们得到一个新的 `sessionID`。其次，也是实现非阻塞的核心，是调用 `client.session.promptAsync()` 来启动任务。我们将新创建的 `sessionID`、要使用的 `agent` 名称以及包含任务描述的 `parts` 数组作为参数传递给 `promptAsync`。如前所述，`promptAsync` 会立即返回一个 204 响应，不会阻塞 `execute` 函数的执行。此时，后台任务已经在 OpenCode 的调度下开始运行，而 `execute` 函数可以继续执行下一步：记录任务状态。

##### 3.2.2.3 监听 `session.idle` 和 `message.part.updated` 事件

为了追踪通过 `promptAsync` 启动的后台任务，插件必须注册一个 `event` Hook 来监听 OpenCode 的事件流。在事件处理函数中，我们需要过滤出与我们启动的后台任务相关的事件。最主要的事件是 `session.idle`。当插件检测到一个 `session.idle` 事件，并且其 `sessionID` 与我们正在追踪的某个后台任务匹配时，就可以断定该任务已经完成。此时，事件处理函数会触发任务的收尾逻辑。此外，还可以监听 `message.part.updated` 事件来实现对任务进度的实时监控。当一个后台会话中的代理产生新的消息片段时，这个事件会被触发。通过监听它，插件可以获取到后台任务的中间输出，例如，代理正在思考的步骤或正在执行的代码。这对于实现长时间任务的进度反馈非常有用，尽管对于简单的“fire-and-forget”模式并非必需。

##### 3.2.2.4 任务结果持久化与通知

当 `session.idle` 事件处理函数被触发，确认一个后台任务完成后，插件需要执行一系列操作来完成闭环。首先，**获取完整结果**。插件会调用 `client.session.messages({ path: { id: backgroundSessionId } })` API 来获取后台会话的完整消息历史。通过解析这些消息，可以提取出子代理的最终响应文本。其次，**持久化结果**。为了防止结果在上下文压缩中丢失，并允许主会话在任意时刻获取结果，插件应该将提取出的结果保存到本地文件系统中。`opencode-background-agents` 插件采用的模式值得借鉴：为每个项目或主会话创建一个独立的目录（如 `~/.local/share/<plugin-name>/results/<root-session-id>/`），并使用唯一的任务 ID 作为文件名（如 `<delegation-id>.md`）[^1^]。这样，即使主会话的上下文被压缩或重启，结果依然安全地保存在磁盘上。最后，**通知主会话**。为了告知主代理任务已完成，插件可以调用 `client.session.prompt({ path: { id: parentSessionId }, body: { noReply: true, parts: [{ type: "text", text: "<task-complete> Task <id> finished. Use `async_retrieve` to see the result." }] } })`。通过设置 `noReply: true`，这个通知会以系统消息的形式注入到主会话的上下文中，而不会触发主代理产生新的回复，从而避免了不必要的交互循环。

#### 3.2.3 优势分析

##### 3.2.3.1 真正非阻塞，主会话无感知

本方案最显著的优势在于它实现了**真正意义上**的非阻塞执行。通过使用 `Session.promptAsync()`，主会话在启动后台任务的瞬间就完全释放了其执行线程。主代理可以立即继续处理用户的下一个指令，TUI 也可以保持对用户输入的响应。这与 `opencode-background-agents` 那种虽然非 `await` 但仍然依赖 `session.prompt()` 的内部机制相比，在理论上更加纯粹和可靠，因为它直接使用了 OpenCode 官方提供的异步 API [^31^]。这种设计确保了无论后台任务执行多久，主会话的交互体验都不会受到影响，从而完美达成了“用户无感”的设计目标。

##### 3.2.3.2 与 OMO-Slim 等插件互不干扰

该方案的另一个核心优势是其**高度的独立性和共存性**。由于它创建了一套全新的、以 `async_` 为前缀的工具，它并没有试图修改或拦截 OMO-Slim 所使用的原生 `task` 工具。OMO-Slim 可以继续其原有的工作方式，而我们的插件则在旁边提供一个并行的异步通道。两者之间不存在直接的依赖或冲突。只要 OpenCode 的插件系统能够同时加载和管理多个插件的工具和事件处理器，这个方案就能稳定工作。这种“在 OMO-Slim 的盲区建立独立生态”的思路，实现了物理隔离级别的抗干扰能力，是解决共存问题的最优雅方式。

##### 3.2.3.3 利用原生 API，兼容性好

本方案完全基于 OpenCode 官方 SDK 提供的标准 API，如 `Session.create()`、`Session.promptAsync()` 和 `event` Hook。它没有使用任何非官方的、hacky 的或未公开的技术。这意味着该插件具有良好的向前兼容性，能够随着 OpenCode 平台的更新而稳定运行。同时，由于它遵循了 OpenCode 的标准插件开发范式，其代码也更容易被其他开发者理解和维护。一个相关的 GitHub 功能请求提案也明确指出了这种基于 `promptAsync` 和 SSE 的插件方案是“零上游代码变更”（zero core code changes）的，强调了其作为现有功能之上的一层“薄分发层”（thin distribution layer）的优势 [^31^]。

#### 3.2.4 潜在挑战

##### 3.2.4.1 `promptAsync` 的稳定性与官方支持程度

尽管 `promptAsync` API 存在于 OpenCode SDK 中，但其稳定性和官方支持程度可能是一个潜在的风险点。与广泛使用的 `Session.prompt()` 相比，`promptAsync` 的使用场景可能相对较少，因此可能存在一些未知的 Bug 或在特定情况下的不稳定行为。一个相关的 GitHub issue 提到，尽管 `promptAsync` 的返回时间非常快（2-4ms），但完整的生成过程仍然可能存在显著的延迟，这表明 `promptAsync` 只是将等待过程从同步调用转移到了后台，而核心的生成延迟问题依然存在 [^37^]。对于我们的异步插件而言，这本身不是问题，因为我们正是需要它不等待。但我们必须密切关注 OpenCode 的更新日志和社区反馈，以确保 `promptAsync` 的行为符合预期，并且在未来的版本中不会被弃用或更改。

##### 3.2.4.2 事件监听的可靠性与延迟

整个异步流程的可靠性严重依赖于 SSE 事件流的稳定性和事件触发的及时性。如果 `session.idle` 事件由于某种原因（例如，OpenCode 内部的 Bug 或网络问题）没有被触发，或者被延迟触发，那么插件将无法正确地感知任务的完成状态，导致任务结果无法被及时保存和通知。同样，如果事件流连接意外断开，插件可能会错过关键事件。因此，在实现中需要考虑健壮的事件监听机制，例如，实现断线重连逻辑，并可能需要设置一个备用的超时机制。插件可以为每个启动的任务设置一个定时器，如果在预定时间内没有收到 `session.idle` 事件，就认为任务可能已经失败或超时，并进行相应的处理。这种“事件驱动 + 超时保险”的双保险策略可以提高系统的整体可靠性。

### 3.3 方案二：基于 `session.prompt()` (非 await) 的 Fire-and-Forget 插件

#### 3.3.1 核心思想：借鉴 `opencode-background-agents` 的实现

本方案的核心思想是直接借鉴 `opencode-background-agents` 插件的成功实践，采用一种基于 JavaScript 异步特性的“fire-and-forget”模式。该模式的核心在于，虽然启动后台任务仍然调用的是看似同步的 `Session.prompt()` API，但通过**不显式地 `await` 其返回的 Promise**，使得代码的执行流程可以立即继续，从而在效果上实现了非阻塞。这种方法绕过了对 `promptAsync` API 的依赖，可能在一些 `promptAsync` 支持不完美或行为不确定的环境中具有更好的兼容性。同时，它也复用了一个经过社区验证的、相对成熟的解决方案，降低了从头设计和实现的风险。

#### 3.3.2 实现流程

##### 3.3.2.1 插件注册自定义工具

与方案一类似，第一步也是通过插件的 `tool` Hook 注册一个自定义的异步委派工具，例如 `fire_and_forget`。这个工具将作为主代理启动此类任务的入口。其参数和返回值的定义与 `async_delegate` 类似，主要接收 `prompt` 和 `agent` 等参数，并返回一个任务 ID 用于后续追踪。这个工具的存在，使得主代理可以明确地区分何时应该使用这种“fire-and-forget”模式，而不是原生的阻塞式 `task` 或方案一中的 `promptAsync` 模式。

##### 3.3.2.2 工具内部创建会话并 `prompt()`，但不 `await`

在 `fire_and_forget` 工具的 `execute` 函数中，实现流程如下：首先，通过 `await client.session.create()` 创建一个新的后台会话。然后，调用 `client.session.prompt()` 来启动子代理任务。这里是关键的一步：我们不使用 `await` 关键字来等待这个 `prompt()` 调用完成。代码会立即继续执行到 `execute` 函数的末尾，并返回一个包含任务 ID 的对象。与此同时，那个未被等待的 `prompt()` Promise 在后台继续执行。为了确保错误能够被捕获，我们会在这个 Promise 上链式调用一个 `.catch()` 方法，用于在后台任务失败时记录日志或更新内部任务状态。

```javascript
// 伪代码示例
const session = await client.session.create({ body: { title: "Task: " + taskId, parentID: ctx.sessionID } });
const promptPromise = client.session.prompt({
  path: { id: session.data.id },
  body: { agent: input.agent, parts: [{ type: "text", text: input.prompt }] }
});

// 不 await promptPromise，而是附加错误处理
promptPromise.catch(error => {
  console.error(`Background task ${taskId} failed:`, error);
  this.taskManager.updateTaskStatus(taskId, 'failed', error.message);
});

// 立即返回任务 ID
return { task_id: taskId, status: 'started' };
```

##### 3.3.2.3 依赖 `session.idle` 事件进行结果处理

与方案一完全相同，任务的完成检测也依赖于对 `session.idle` 事件的监听。插件需要注册 `event` Hook，并在事件处理函数中检查事件来源的 `sessionID` 是否匹配任何正在运行的后台任务。一旦匹配成功，插件就会执行相同的收尾逻辑：获取会话消息、持久化结果、通知主会话。这表明，无论是使用 `promptAsync` 还是非 `await` 的 `prompt`，事件驱动的完成通知机制都是不可或缺的组成部分。

#### 3.3.3 优势分析

##### 3.3.3.1 实现相对简单，复用现有成熟方案

本方案的最大优势在于其**实现相对简单，并且复用了 `opencode-background-agents` 这一经过社区验证的成熟模式**。开发者无需深入研究 `promptAsync` 和 SSE 的复杂交互，只需要理解 JavaScript 的 Promise 和事件循环机制即可。`opencode-background-agents` 的源代码提供了一个完整且高效的实现参考，可以直接借鉴其任务管理、状态追踪和结果通知的逻辑，从而大大缩短开发周期并降低引入新 Bug 的风险。对于那些希望快速实现功能，并且对 `promptAsync` 的稳定性持保守态度的开发者来说，这是一个非常吸引人的选择。

##### 3.3.3.2 无需依赖 `promptAsync` 的可用性

由于本方案完全依赖于标准的 `Session.prompt()` API，它只是巧妙地利用了 JavaScript 的异步特性来改变其使用方式，因此它不依赖于 `Session.promptAsync()` 的可用性或稳定性。这在某些特定版本的 OpenCode 或特定配置下可能是一个优势。如果 `promptAsync` 的行为存在不确定性，或者其文档说明不够清晰，那么采用这种更“传统”的 `prompt()` 方法可能更安全。它基于一个广为人知且稳定的核心 API，只是以一种非传统的方式使用它。

#### 3.3.4 潜在挑战

##### 3.3.4.1 本质上是“hack”，未来 OpenCode 更新可能导致行为变化

本方案的挑战在于其解决方案的“hacky”本质。它并非 OpenCode 官方设计的异步执行方式，而是通过利用 JavaScript 语言特性的一个“副作用”来实现的。这意味着，未来 OpenCode 的更新可能会以某种方式改变 `Session.prompt()` 的内部行为，从而破坏这种非 `await` 的模式。例如，如果 OpenCode 在未来的版本中为 `prompt()` 添加了某种强制性的、必须在调用端等待的机制，那么这种“fire-and-forget”的技巧将不再有效。虽然这种可能性不大，但它确实引入了一种与未来版本不兼容的潜在风险。

##### 3.3.4.2 对未捕获的 Promise 错误处理需要格外小心

当一个 Promise 没有被 `await` 时，如果它最终以 `rejected` 状态结束（即发生错误），并且没有附加 `.catch()` 处理函数，它就会成为一个“未处理的 Promise rejection”。在较新版本的 Node.js 中，这可能会导致进程打印警告信息，甚至在某些配置下可能导致进程终止。因此，在使用这种模式时，必须**极其小心地处理所有可能的错误情况**。必须为每一个未被等待的 `prompt()` Promise 附加一个 `.catch()` 处理器，以确保任何错误都能被优雅地捕获和处理，避免产生未处理的 rejection。这增加了代码的复杂性，并要求开发者对异步错误处理有深刻的理解。

### 3.4 方案三：基于进程隔离的 Out-of-Process 子代理

#### 3.4.1 核心思想：完全脱离 OpenCode 会话树，通过子进程运行任务

本方案提出了一个更为激进的思路：彻底脱离 OpenCode 的会话管理和子代理机制，通过在插件中直接启动一个完全独立的子进程（例如，通过 Node.js 的 `child_process` 模块）来运行后台任务。这个子进程可以是一个独立的脚本，它直接使用 LLM 提供商的 API（如 OpenAI API）来完成任务，而不是通过 OpenCode 的代理系统。这种方式实现了最高级别的隔离，因为后台任务不再运行在任何 OpenCode 会话中，它是一个完全外部的、独立的进程。这从根本上避免了与 OpenCode 内部会话管理、事件循环和撤销历史的任何冲突。

#### 3.4.2 实现流程

##### 3.4.2.1 插件内部通过 `child_process` 等模块启动新进程

在 `execute` 函数中，当需要启动一个后台任务时，插件不再调用任何 OpenCode 的会话相关 API。相反，它会使用 Node.js 的 `child_process.spawn()` 或 `child_process.fork()` 方法来启动一个新的 Node.js 进程。这个子进程将执行一个预设的脚本文件（例如 `runner.js`）。主进程（插件）可以通过环境变量或标准输入（stdin）将任务所需的参数（如任务描述、LLM 模型配置等）传递给子进程。同时，主进程可以监听子进程的 `stdout` 和 `stderr` 来获取任务的实时输出和错误信息。

##### 3.4.2.2 新进程直接使用 LLM API 完成任务

在子进程 `runner.js` 中，它将直接调用 LLM 提供商的 API。例如，它可以使用 `openai` 或 `@anthropic-ai/sdk` 等客户端库，根据从父进程接收到的任务描述，构造一个 API 请求并发送。子进程将直接处理 LLM 的响应流，并将最终的执行结果或中间状态通过 `stdout` 或写入到一个约定的文件中，与父进程进行通信。这种方式完全绕过了 OpenCode 的模型路由、工具调用和会话管理机制，将后台任务变成了一个纯粹的、独立的 LLM 交互过程。

##### 3.4.2.3 通过文件系统或进程间通信 (IPC) 传递结果

父进程（插件）需要通过某种机制来获取子进程的执行结果。最简单的方式是通过文件系统。子进程在完成任务后，将结果写入到一个由父进程指定的临时文件中，然后退出。父进程通过监听子进程的 `close` 事件来得知任务已结束，然后读取该临时文件获取结果。另一种更高效的方式是使用 Node.js 的进程间通信（IPC）机制。如果在 `child_process.fork()` 中设置了 `stdio: ['pipe', 'pipe', 'pipe', 'ipc']`，父进程和子进程之间就可以通过 `process.send()` 和 `process.on('message', ...)` 来直接发送和接收消息，从而实现更灵活的通信。

#### 3.4.3 优势分析

##### 3.4.3.1 最高级别的隔离，完全不受 OpenCode 会话机制影响

这是本方案最无与伦比的优势。由于后台任务运行在一个完全独立的进程中，并且直接与 LLM API 交互，它与 OpenCode 的内部世界没有任何联系。这意味着它**完全不受** OpenCode 的会话树、撤销系统、上下文压缩、插件冲突等任何内部机制的影响。这种方案提供了最高级别的健壮性和可预测性，是实现真正“物理隔离”的异步执行的唯一途径。

##### 3.4.3.2 可实现写操作，无撤销历史冲突

与 `opencode-background-agents` 只能进行只读操作的限制不同，由于完全脱离了 OpenCode 的撤销系统，本方案中的后台任务可以**自由地执行任何写操作**，例如修改代码文件、执行 `git` 命令等，而不会有任何破坏撤销历史的担忧。这使得该方案适用于更广泛的后台任务场景，包括那些需要实际修改项目状态的自动化构建、测试或代码重构任务。

#### 3.4.4 潜在挑战

##### 3.4.4.1 实现复杂度高，需要独立管理 LLM 调用和上下文

本方案的最大挑战在于其**极高的实现复杂度**。开发者需要自己管理 LLM 的 API 调用、错误处理、重试逻辑、流式响应解析等，而这些功能在 OpenCode 内部已经得到了很好的实现。此外，由于脱离了 OpenCode 的会话系统，开发者还需要自己实现上下文的维护和传递。如果后台任务需要多轮交互或复杂的工具调用循环，那么在子进程中独立实现这些逻辑将是一项非常艰巨的任务。这实际上是在 OpenCode 之外重新构建了一个微型的代理执行框架。

##### 3.4.4.2 无法利用 OpenCode 的代理配置、工具（如 bash, edit）和上下文管理

由于子进程是独立运行的，它将**无法利用**用户为 OpenCode 配置的任何代理设置，如模型选择、系统提示词、温度等。它也无法使用 OpenCode 提供的强大工具，如安全的 `bash` 执行环境、基于 LSP 的 `edit` 工具等。这使得后台任务的能力大大受限，其执行环境可能远不如在 OpenCode 会话中运行的子代理。这种方案牺牲了大量的现有功能和便利性，以换取极致的隔离性。

##### 3.4.4.3 进程管理和资源消耗

启动和管理独立的子进程会带来额外的系统资源消耗。每个后台任务都会占用一个独立的进程，如果并发任务很多，可能会对系统性能产生影响。此外，开发者还需要自己实现健壮的进程管理逻辑，包括处理进程的异常退出、僵尸进程、资源泄漏等问题。这增加了系统的运维复杂性和不稳定性。

### 3.5 方案四：插件级 Hook 拦截与代理

#### 3.5.1 核心思想：拦截 `task` 工具调用，内部转换为异步执行

本方案提出了一种更具侵入性的思路：通过插件的 `tool.execute.before` Hook，拦截所有对原生 `task` 工具的调用。在拦截后，插件不会阻止该调用，而是会“劫持”它，将原本 intended for 同步执行的 `task` 调用，在插件内部偷偷地转换为我们自己实现的异步执行流程（例如，使用方案一或方案二的机制）。从主代理的角度看，它认为自己只是发起了一个普通的 `task` 调用，但实际上，这个调用在底层被我们的插件以异步、非阻塞的方式处理了。这种方案试图在不改变主代理行为模式的情况下，“透明地”解决阻塞问题。

#### 3.5.2 实现流程

##### 3.5.2.1 注册 `tool.execute.before` Hook

首先，插件需要在 `tool.execute.before` Hook 中注册一个处理函数。这个函数会在任何工具执行前被调用。

##### 3.5.2.2 检测到对 `task` 工具的调用时，阻止其执行

在 Hook 处理函数中，检查 `input.toolName`。如果它等于 `'task'`，就意味着一个原生任务委派即将发生。此时，我们将 `output.proceed` 设置为 `false`，这会阻止原生的 `task` 工具被执行。

##### 3.5.2.3 在 Hook 内部启动一个异步任务，并立即返回一个“模拟”的结果

阻止原生 `task` 执行后，Hook 处理函数将接管控制权。它会提取 `input.arguments` 中的参数（如 `prompt` 和 `agent`），然后调用我们自己实现的异步任务启动函数（例如，`asyncDelegateInternal(...)`）。这个内部函数会按照方案一或方案二的流程，通过 `promptAsync` 或非 `await` 的 `prompt` 来启动一个真正的后台任务。最后，Hook 处理函数会构造一个“模拟”的 `task` 工具执行结果，并将其返回。这个模拟结果可以是一个简单的字符串，如 `"Task 'xyz' has been dispatched to run in the background."`，从而让主代理认为 `task` 调用已经成功完成，可以继续下一步。

```javascript
// 伪代码示例
"tool.execute.before": async (input, output) => {
  if (input.toolName === 'task') {
    output.proceed = false; // 阻止原生 task 执行
    
    // 启动我们自己的异步任务
    const taskId = await asyncDelegateInternal(input.arguments);
    
    // 返回一个模拟的成功结果
    output.result = `Task '${taskId}' has been dispatched asynchronously.`;
  }
}
```

#### 3.5.3 优势分析

##### 3.5.3.1 对主代理透明，无需修改其调用逻辑

本方案的主要优势在于其**对上层代理的透明性**。主代理（包括 OMO-Slim 的编排器）仍然按照其原有的逻辑调用 `task` 工具，它不需要知道底层已经换成了异步实现。这意味着该方案可以与任何使用 `task` 工具的现有插件或代理兼容，无需修改它们的代码。这在理论上提供了一种“一劳永逸”的解决方案，可以透明地提升整个 OpenCode 生态的异步能力。

#### 3.5.4 潜在挑战

##### 3.5.4.1 高度依赖 Hook 的执行顺序和稳定性

`tool.execute.before` Hook 可能会有多个插件同时注册。Hook 的执行顺序可能会影响本方案的正常工作。如果有其他插件在我们的插件之前拦截了 `task` 调用并改变了其行为，我们的插件可能就无法正确工作。此外，过度依赖 Hook 机制会引入不稳定性，因为任何 Hook 处理函数中的错误都可能导致整个工具调用流程失败。

##### 3.5.4.2 可能导致不可预期的副作用，调试困难

“劫持”原生工具的行为是一种非常规的做法，很容易引入难以预料的副作用。例如，主代理可能期望 `task` 调用返回一个包含子代理详细执行结果的字符串，但我们返回的只是一个简单的成功通知。这可能会导致主代理的后续逻辑出错，因为它没有得到期望的数据。此外，当出现问题时，调试会变得非常困难，因为主代理、我们的插件和 OpenCode 核心之间的交互变得非常复杂和隐晦。这种方案的“魔法”特性使其在生产环境中使用时风险很高。

### 4.2 综合评估与推荐

#### 4.2.1 推荐方案：方案一（`promptAsync` + SSE）

综合以上对比矩阵，**我们强烈推荐采用方案一：基于 `promptAsync` + SSE 的独立异步调度插件**。该方案在非阻塞保证、与现有插件的兼容性、隔离性、以及对 OpenCode 生态的利用等多个维度上都取得了最佳的平衡。它使用了 OpenCode 官方提供的、为异步场景设计的 `promptAsync` API，从机制上保证了非阻塞性。通过创建一套独立的 `async_` 工具集，它实现了与 OMO-Slim 等插件的物理隔离，避免了任何潜在的冲突。同时，它完全基于 OpenCode SDK 的标准用法，具有良好的稳定性和向前兼容性。虽然其 `promptAsync` 的稳定性是一个潜在风险，但作为一个官方 API，其被长期支持的可能性远高于方案二的非官方“hack”。

#### 4.2.2 理由：平衡了非阻塞性、兼容性、实现复杂度和可维护性

选择方案一的核心理由在于其在各个关键指标之间取得了最佳平衡。与方案二相比，它使用了更官方、更可靠的异步机制，降低了未来兼容性风险。与方案三相比，它极大地降低了实现复杂度和维护成本，并能充分利用 OpenCode 强大的代理和工具生态系统，避免了“重复造轮子”。与方案四相比，它提供了更好的隔离性和可预测性，避免了因 Hook 冲突或副作用而导致的难以调试的问题。方案一提供了一条清晰、稳健且功能强大的路径，能够在满足用户核心需求（非阻塞、无感、共存）的同时，保持代码的整洁和可维护性。

### 4.3 与 OMO-Slim 的共存策略

#### 4.3.1 独立工具集，避免命名冲突

实现与 OMO-Slim 等插件和平共存的关键策略是**物理隔离**。我们的插件将注册一套以 `async_` 为前缀的、独一无二的工具集，例如 `async_delegate`、`async_retrieve`、`async_list`、`async_cancel` 等。这些工具的命名空间与 OMO-Slim 可能使用的 `task` 或 `delegate_task` 等工具完全不重叠。主代理在使用时，必须明确调用我们的 `async_delegate` 工具才能触发异步流程。这种设计避免了任何直接的功能冲突或命名空间污染，确保了插件之间的独立性。

#### 4.3.2 通过 `event` Hook 监听全局事件，但仅处理自身创建的任务

在事件监听方面，我们的插件会注册 `event` Hook 来监听全局的 `session.idle` 等事件。然而，在事件处理函数内部，我们会进行严格的过滤。插件内部将维护一个数据结构（如一个 `Map`），用于记录所有由其自身通过 `async_delegate` 启动的后台任务的 `sessionID`。当接收到一个事件时，处理函数会首先检查事件的 `sessionID` 是否存在于这个内部记录中。只有当事件属于我们自己的任务时，我们才会对其进行处理（如触发结果持久化和通知逻辑）。对于其他插件（如 OMO-Slim）创建的任务所产生的事件，我们的插件将完全忽略它们。这种“监听全局，但只处理私有”的策略，确保了我们的异步逻辑不会意外地干扰到其他插件的正常工作。

#### 4.3.3 提供清晰的文档，指导用户在需要异步场景时使用本插件的工具

为了确保用户能够正确地使用我们的插件，并提供明确的指导，帮助他们在合适的场景下选择正确的工具。文档应该清晰地解释原生 `task` 工具的阻塞特性，以及我们的 `async_delegate` 工具的非阻塞优势。通过提供明确的对比和使用场景示例，我们可以引导用户或主代理在需要进行长时间后台处理时，主动选择我们的异步工具，从而在应用层面实现了与 OMO-Slim 等插件的无缝协作和共存。

## 5. 推荐方案的详细设计与实现要点

### 5.1 插件结构

#### 5.1.1 `tool` Hook：注册 `async_delegate`, `async_retrieve`, `async_list` 工具

插件的核心入口将是一个导出的异步函数，它返回一个实现了 `Plugin` 接口的对象。在这个对象中，`tool` 属性将是一个函数，该函数返回一个工具字典。这个字典将包含我们自定义的三个核心工具。每个工具都将使用 OpenCode 插件 SDK 提供的 `tool()` 函数来定义，该函数接收一个描述对象，包含 `description`（工具的功能描述，将被 LLM 用于决策何时调用该工具）、`parameters`（工具的参数模式，使用 Zod schema 定义）和 `execute`（工具被调用时执行的异步函数）。

```typescript
import { Plugin, tool } from '@opencode-ai/plugin';
import { z } from 'zod';

export const StandaloneAsyncPlugin: Plugin = async (ctx) => {
  const taskManager = new AsyncTaskManager(ctx.client); // 核心任务管理器

  return {
    tool: () => ({
      async_delegate: tool({
        description: 'Dispatches a task to a sub-agent to be executed asynchronously in the background. Returns a task ID immediately without blocking.',
        parameters: z.object({
          prompt: z.string().describe('The detailed task description for the sub-agent.'),
          agent: z.string().describe('The name of the sub-agent to handle the task.'),
        }),
        execute: async (args, ctx) => {
          return taskManager.delegate(args, ctx.sessionID, ctx.messageID);
        },
      }),
      async_retrieve: tool({
        description: 'Retrieves the result of a completed asynchronous task by its ID. Blocks until the task is finished if it is still running.',
        parameters: z.object({
          task_id: z.string().describe('The unique ID of the async task to retrieve.'),
        }),
        execute: async (args, ctx) => {
          return taskManager.retrieve(args.task_id);
        },
      }),
      async_list: tool({
        description: 'Lists all active and recently completed asynchronous tasks.',
        parameters: z.object({}),
        execute: async () => {
          return taskManager.list();
        },
      }),
    }),
    event: async ({ event }) => {
      // 事件处理逻辑
      if (event.type === 'session.idle') {
        await taskManager.handleSessionIdle(event.properties.sessionID);
      }
    },
  };
};
```

#### 5.1.2 `event` Hook：监听 `session.idle` 事件

除了 `tool` Hook，插件还会注册一个 `event` Hook。这个 Hook 是一个异步函数，它接收一个包含 `event` 对象参数。在这个函数中，我们将实现对特定事件的监听和处理。最关键的监听对象是 `session.idle` 事件。当 OpenCode 发布一个 `session.idle` 事件时，我们的 `event` 处理函数会被调用。函数内部会检查该事件的 `sessionID` 是否与我们 `AsyncTaskManager` 中正在追踪的任何一个后台任务相匹配。如果匹配，就会调用 `taskManager.handleSessionIdle(sessionID)` 方法来触发任务的完成处理流程。

#### 5.1.3 内部核心逻辑：`AsyncTaskManager` 类

为了保持代码的清晰和可维护性，插件的核心业务逻辑将被封装在一个名为 `AsyncTaskManager` 的类中。这个类将负责管理所有异步任务的生命周期，包括任务的创建、状态追踪、完成检测、结果持久化和通知。在插件的入口函数中，我们会实例化这个类，并将其注入到工具的 `execute` 函数和事件处理函数中。`AsyncTaskManager` 的实例将持有对 OpenCode SDK 客户端（`ctx.client`）的引用，以便调用 `session.create`、`session.promptAsync` 等 API。它还将维护一个内部的数据结构（例如 `tasks: Map<string, TaskInfo>`），用于存储当前所有活跃任务的状态信息。

### 5.2 核心逻辑实现

#### 5.2.1 `AsyncTaskManager.delegate()`：创建任务，调用 `promptAsync`，返回任务 ID

`delegate` 方法是启动异步任务的入口。它的实现流程如下：
1.  **生成唯一任务 ID**：使用 `uuid` 或类似的库生成一个唯一的任务 ID。
2.  **创建后台会话**：调用 `this.client.session.create({ body: { title: `Async Task: ${taskId}`, parentID: parentSessionID } })` 来创建一个新的子会话。
3.  **记录任务信息**：将任务的初始信息（ID、状态 `'running'`、后台会话 ID `backgroundSessionID`、父会话 ID `parentSessionID` 等）记录到内部的 `this.tasks` Map 中。
4.  **启动异步任务**：调用 `this.client.session.promptAsync({ path: { id: backgroundSessionID }, body: { agent: args.agent, parts: [{ type: "text", text: args.prompt }] } })`。注意，这里不需要 `await`，因为它返回的是 204。
5.  **返回任务 ID**：立即返回一个对象 `{ task_id: taskId, status: 'running' }`。

#### 5.2.2 `AsyncTaskManager.handleSessionIdle()`：事件处理，获取结果，持久化，通知

`handleSessionIdle` 方法是对 `session.idle` 事件的响应。它的实现流程如下：
1.  **查找任务**：在 `this.tasks` 中查找与 `idleSessionID` 匹配的任务。
2.  **获取结果**：如果找到任务，调用 `this.client.session.messages({ path: { id: idleSessionID } })` 来获取后台会话的完整消息历史。然后解析这些消息，提取出子代理的最终响应文本。
3.  **更新任务状态**：将任务的状态更新为 `'completed'`，并将提取到的结果保存到任务信息中。
4.  **持久化结果**：将最终结果写入到本地文件系统中，例如 `fs.writeFile(path.join(this.resultsDir, `${taskId}.md`), result, 'utf8')`。
5.  **通知父会话**：调用 `this.client.session.prompt({ path: { id: task.parentSessionID }, body: { noReply: true, parts: [{ type: "text", text: `<async-task-complete> Task ${taskId} is finished. Result saved to ${resultPath}.` }] } })` 向父会话发送一个静默通知。
6.  **清理**：可选地，在通知完成后，可以调用 `this.client.session.delete({ path: { id: idleSessionID } })` 来删除已经完成的后台会话，以保持会话树的整洁。

#### 5.2.3 `AsyncTaskManager.retrieve()`：从文件系统读取并返回结果

`retrieve` 方法用于获取已完成的任务结果。它的实现流程如下：
1.  **查找任务**：在 `this.tasks` 中查找与 `taskId` 匹配的任务。
2.  **检查状态**：如果任务存在且状态为 `'running'`，则进入等待状态。可以使用一个 Promise 和一个超时机制来等待任务完成。例如，可以设置一个轮询或事件监听器，在 `handleSessionIdle` 中将任务标记为完成时，resolve 这个 Promise。
3.  **读取结果**：一旦任务状态变为 `'completed'`，从持久化的文件中读取结果，例如 `const result = await fs.readFile(path.join(this.resultsDir, `${taskId}.md`), 'utf8')`。
4.  **返回结果**：将读取到的结果文本返回。

### 5.3 错误处理与边界情况

#### 5.3.1 后台任务超时处理

为了防止后台任务因各种原因（如陷入死循环、网络问题）而永不结束，我们需要实现一个超时机制。在 `delegate` 方法中启动任务后，可以设置一个 `setTimeout`。如果在指定的时间内（例如 10 分钟）任务没有完成（即没有收到 `session.idle` 事件），`setTimeout` 的回调函数就会被触发。在回调函数中，我们将任务状态更新为 `'timeout'`，并执行清理操作，例如尝试调用 `this.client.session.abort({ path: { id: backgroundSessionID } })` 来强制终止后台会话，并通知用户任务已超时。

#### 5.3.2 后台任务失败处理

后台任务在执行过程中可能会失败，例如，子代理调用了一个不存在的工具，或者代码执行出错。当这种情况发生时，子代理可能会生成一个错误消息，然后会话变为 `idle`。在我们的 `handleSessionIdle` 方法中，除了提取正常结果，还需要能够识别出错误情况。可以通过分析子代理的最后一条消息来判断任务是否成功。如果检测到失败，我们将任务状态更新为 `'failed'`，将错误信息持久化，并通知父会话任务失败的原因。

#### 5.3.3 主会话退出时的后台任务生命周期管理

当用户关闭主会话时，与之关联的后台任务应该如何处理是一个需要考虑的问题。一种策略是“孤儿”任务继续运行，直到它们自己完成或超时。另一种策略是在检测到主会话被删除（`session.deleted` 事件）时，主动清理所有与之关联的后台任务。这可以通过在 `event` Hook 中监听 `session.deleted` 事件来实现。当收到该事件时，遍历 `this.tasks`，找到所有 `parentSessionID` 匹配的任务，并调用 `this.client.session.delete()` 来终止它们。选择哪种策略取决于具体的业务需求，但显式的清理通常是更健壮的做法。
