# Aristotle Checker Protocol

> Loaded by Checker subagent (C role) and Coordinator during post-hoc review.
> Do NOT load into parent session.

C role responsibilities: schema compliance, content accuracy, scope determination, rule writing to Git.

---

## Parameters (Subagent Mode)

| Parameter | Description |
|-----------|-------------|
| `DRAFT_SEQUENCE` | State record sequence number |
| `DRAFT_FILE` | Path to persisted DRAFT file |
| `PROJECT_DIRECTORY` | Project directory (scope determination) |

---

## STEP C1: LOAD DRAFT

```
read DRAFT_FILE
  → fail? output "❌ DRAFT file not found at [path]. Aborting." STOP
  → ok?  parse Reflections (each starts with "### Reflection N:")
```

---

## STEP C2: SCHEMA COMPLIANCE

```
for each Reflection verify:
  category      ∈ 8 valid categories
  intent_tags   → domain + task_goal non-empty
  error_summary → present and ≤200 chars

auto-correct (trivial defaults only, mark [schema-auto-fixed]):
  missing/invalid category → SYNTAX_API_ERROR
  missing domain           → "general"
  missing task_goal         → "unspecified"
  overlong error_summary   → truncate to 200 chars

if field requires judgment beyond trivial defaults → leave, mark for staging review
id is auto-generated by write_rule — do NOT validate
```

---

## STEP C3: CONTENT ACCURACY

Cross-reference structured fields against Incident block. Checks 1–5 critical (stop and flag). Checks 6–8 structural (auto-fix if possible).

```
critical checks:
  1. category accuracy
     hallucination → HALLUCINATION (not INCOMPLETE_ANALYSIS)
     tool misuse   → WRONG_TOOL_CHOICE (not SYNTAX_API_ERROR)
     misunderstood → MISUNDERSTOOD_REQUIREMENT (not ASSUMED_CONTEXT)

  2. intent tags accuracy
     domain/task_goal match what user was actually doing
     e.g. error during database migration → domain should be database_operations not code_generation
     task_goal = user's original intent, not the error

  3. error summary quality
     concise, accurate, focused on what went wrong (not why)
     specific enough to distinguish from other errors, consistent with Incident

  4. failed skill accuracy
     reasoning mistake with no tool → null

  5. proposed rule quality
     addresses root cause from 5-Why
     specific and actionable (not "be more careful")
     verifiable: clear ✅/❌ test case

structural checks:
  6. context + example consistency
     Context = trigger conditions for rule
     Example ✅ = behavior that would prevent error
     Example ❌ = matches actual error

  7. incident structure
     User Request: present, non-empty, ≤400 chars
     Model Wrong Output: present, non-empty, ≤400 chars
     User Correction OR Error Impact: at least one present
     User Correction ≤150 chars, Error Impact ≤100 chars
     total Incident ≤1000 chars (preserve Model Wrong Output → truncate User Request → truncate optional fields)

  8. incident quality
     User Request = actual task, not "user made a request"
     Model Wrong Output = actual error, not "model gave wrong code"
     overly generalized → flag [NEEDS_SPECIFIC_INCIDENT], reject

legacy compat (check FIRST before structural checks 7–8):
  DRAFT contains Error Excerpt / Correction Excerpt (old format) instead of Incident:
    Error Excerpt → treat as Model Wrong Output
    Correction Excerpt → treat as User Correction
    User Request inferred from Error Excerpt context
```

---

## STEP C4: SCOPE DETERMINATION

```
determine scope by error nature (NOT project path):

  model reasoning failure / generic tool misuse → user (root cause is cross-project; rule applies universally)
  project-specific convention / tech stack       → project (rule only makes sense in this project)
  cannot determine                               → user (conservative default — user-level rules are safer)

analyze DRAFT's 5-Why Root Cause, Incident, and Intent Tags to make this determination.
if scope="project": set project_path = PROJECT_DIRECTORY
```

### DRAFT → Rule Body Format

```
DRAFT (nested):                    Rule body (flat):
- **Incident**:                    ### Incident
  - **User Request:** ...          **User Request:** ...
  - **Model Wrong Output:** ...    **Model Wrong Output:** ...

Checker performs this transformation in write_rule call.
```

---

## STEP C5: WRITE RULES TO GIT

> Constraint: Do NOT use `skill_mcp`. Call `aristotle_*` tool functions directly.

```
for each validated Reflection:
  1. aristotle_write_rule(
       content, scope, category, source_session, message_range,
       intent_domain, intent_task_goal, failed_skill, error_summary,
       rule_summary=<Proposed Rule text from DRAFT Reflection>,
       project_path, confidence=0.7)
     → returns file_path, rule_id

  2. aristotle_stage_rule(file_path)
     → status = "staging"

  3. aristotle_get_audit_decision(file_path)
     → returns delta, audit_level

  4. if audit_level == "auto":
       aristotle_commit_rule(file_path) → committed to Git
     if "semi" or "manual":
       leave staging → user reviews via /aristotle review N
```

---

## STEP C6: OUTPUT RESULT

```
CHECKER RESULT:
- Rules processed: N
- Auto-committed: X
- Staged for review: Y
- Schema corrections: Z
- Scope decisions: [list each rule's scope and rationale]

STOP. Coordinator reads output and notifies user.
```

---

## POST-HOC REVIEW MODE

```
all checks pass    → proceed to commit
schema failure     → auto-correct trivial, then commit
content inaccuracy (auto-correct if high confidence, otherwise flag to user):
  category mismatch → auto-correct, show correction
  vague summary     → auto-improve, show improvement
  generic rule      → flag to user, suggest specific wording
soft warning        → note in output, proceed
```
