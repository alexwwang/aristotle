# Aristotle Checker Protocol

> Loaded by the Checker subagent (C role) during automated flow,
> and by the Coordinator during post-hoc review validation.
> Do NOT load this into the parent session context.

This file defines the C (Checker) role's responsibilities:
1. **Schema compliance** — structural validation of each Reflection
2. **Content accuracy** — semantic validation against original error
3. **Scope determination** — user vs project based on error nature
4. **Rule writing** — serialize validated Reflections to Git via MCP

---

## Parameters (Subagent Mode)

When fired as a subagent, the Coordinator passes:
- `DRAFT_SEQUENCE` — State record sequence number
- `DRAFT_FILE` — Path to the persisted DRAFT file
- `PROJECT_DIRECTORY` — Project directory (for scope determination)

---

## STEP C1: LOAD DRAFT (Subagent Mode)

Read the DRAFT file from `DRAFT_FILE` path.

1. Use the Read tool to read the file at `DRAFT_FILE`
2. If the file cannot be read, output: "❌ DRAFT file not found at [DRAFT_FILE]. Aborting."
3. Parse the DRAFT to extract individual Reflections (each starts with `### Reflection N:`)

---

## STEP C2: SCHEMA COMPLIANCE

For each Reflection, verify structural correctness:

- `category` — present and is one of the 8 valid categories
- `intent_tags` — present with both `domain` and `task_goal` non-empty
- `error_summary` — present and ≤200 characters

Auto-correct trivial failures (limited to defaults only):

- Missing/invalid category → set to `SYNTAX_API_ERROR` and mark `[schema-auto-fixed]`
- Missing `domain` → set to `"general"`
- Missing `task_goal` → set to `"unspecified"`
- Overlong `error_summary` → truncate to 200 characters

**Do NOT attempt complex inference for auto-correction.** If a field requires judgment beyond trivial defaults, leave it and mark the Reflection for staging review.

> **Note:** `id` is auto-generated by `write_rule` MCP tool (format: `rec_{timestamp}`). Checker does not validate it — it doesn't exist until the MCP call.

---

## STEP C3: CONTENT ACCURACY

Cross-reference each Reflection's structured fields against the Incident block (User Request, Model Wrong Output, User Correction) in the DRAFT:

> **Check priority:** Checks 1–5 are critical — stop and flag on failure. Checks 6–8 are structural — auto-fix if possible.

1. **Category accuracy** — Does the chosen category match the actual error type?
   - Error is a hallucination (model made up code/facts) → should be `HALLUCINATION`, not `INCOMPLETE_ANALYSIS`
   - Error is a tool misuse → should be `WRONG_TOOL_CHOICE`, not `SYNTAX_API_ERROR`
   - Error is a requirement misunderstanding → should be `MISUNDERSTOOD_REQUIREMENT`, not `ASSUMED_CONTEXT`

2. **Intent tags accuracy** — Do `domain` and `task_goal` match what the user was actually doing?
   - Cross-check against Incident's User Request and Model Wrong Output: if the error occurred during database migration, `domain` should be `"database_operations"` not `"code_generation"`
   - `task_goal` should describe the user's original intent, not the error itself

3. **Error summary quality** — Is `error_summary` concise, accurate, and focused on what went wrong?
   - Should be specific enough to distinguish from other errors
   - Should be consistent with Incident

4. **Failed skill accuracy** — Does `failed_skill` correctly identify the tool/skill involved?
   - If error is a reasoning mistake with no specific tool → should be `null`

5. **Proposed rule quality** — Is the prevention rule specific and actionable?
   - Must directly address the root cause from the 5-Why analysis
   - Must not be overly generic (e.g., "be more careful" is not a valid rule)
   - Must be verifiable: there should be a clear ✅/❌ test case

6. **Context and Example consistency** — Do Context and Example align with the error scene?
   - Context should specify when the rule applies (trigger conditions)
   - Example ✅ should demonstrate correct behavior that would have prevented the error
   - Example ❌ should match the actual error that occurred

**Legacy format compatibility (check first):** If the DRAFT contains `Error Excerpt` / `Correction Excerpt` (old format) instead of `Incident`, treat Error Excerpt as Model Wrong Output and Correction Excerpt as User Correction. User Request must be inferred from Error Excerpt context for legacy DRAFTs.

7. **Incident structure** — Is the Incident block properly formatted and within length constraints?
    - Verify `**User Request:**` is present and non-empty
    - Verify `**Model Wrong Output:**` is present and non-empty
    - Verify `**User Correction:**` or `**Error Impact:**` is present (at least one of the two optional fields)
    - Verify individual length constraints: User Request ≤400, Model Wrong Output ≤400, User Correction ≤150, Error Impact ≤100
    - Verify **total Incident length ≤1000 chars** (including labels and formatting). If exceeded, preserve Model Wrong Output in full, then truncate User Request, then truncate optional fields.

8. **Incident quality** — Is the Incident specific to the actual error scene, not a generic description?
    - `User Request` must contain the actual task/request, not vague phrases like "user made a request"
    - `Model Wrong Output` must contain the actual erroneous output, not generic summaries like "model gave wrong code"
    - If either field is overly generalized, flag `[NEEDS_SPECIFIC_INCIDENT]` and reject

---

## STEP C4: SCOPE DETERMINATION

For each Reflection, determine scope based on the **error's nature** (NOT project path existence):

| Error Nature | scope | Reasoning |
|-------------|-------|-----------|
| Model reasoning failure (hallucination, misunderstanding, assumed context) | `user` | Root cause is cross-project; rule applies universally |
| Generic tool misuse (not project-specific) | `user` | Tool behavior is the same across projects |
| Project-specific convention violation | `project` | Rule only makes sense in this project's context |
| Project-specific tech stack issue | `project` | Error depends on this project's libraries/frameworks |
| Cannot determine | `user` | Conservative default — user-level rules are safer |

Analyze the DRAFT's 5-Why Root Cause, Incident, and Intent Tags to make this determination.

If `scope = "project"`, set `project_path` to `PROJECT_DIRECTORY`.

---

### DRAFT → Rule Body Format Transformation

When writing the rule from DRAFT, transform the DRAFT's nested Incident format into the rule body's flat heading format:

**DRAFT format (nested bullets):**
```markdown
- **Incident**:
  - **User Request:** ...
  - **Model Wrong Output:** ...
```

**Rule body format (flat headings):**
```markdown
### Incident

**User Request:** ...
**Model Wrong Output:** ...
```

The Checker must perform this transformation when calling `write_rule`.

---

## STEP C5: WRITE RULES TO GIT (Subagent Mode)

> **Constraint:** Do NOT use `skill_mcp` — subagents do not have MCP server access. Call the `aristotle_*` tool functions directly.

For each validated Reflection, execute the following sequence using direct tool function calls:

1. Call `aristotle_write_rule(content=..., scope=..., category=..., source_session=..., message_range=..., intent_domain=..., intent_task_goal=..., failed_skill=..., error_summary=..., project_path=..., confidence=0.7)`
   → Returns `file_path` and `rule_id`

2. Call `aristotle_stage_rule(file_path=<returned path>)`
   → Status becomes `staging`

3. Call `aristotle_get_audit_decision(file_path=<path>)`
   → Returns `delta` and `audit_level`

4. IF `audit_level == "auto"` → call `aristotle_commit_rule(file_path=<path>)` → rule committed to Git

5. IF `audit_level == "semi"` or `"manual"` → leave in staging status. User can review via `/aristotle review N`

---

## STEP C6: OUTPUT RESULT (Subagent Mode)

Output a structured summary:

```
CHECKER RESULT:
- Rules processed: N
- Auto-committed: X
- Staged for review: Y
- Schema corrections: Z
- Scope decisions: [list each rule's scope and rationale]
```

**STOP.** Do not wait for user input. The Coordinator will read your output and notify the user.

---

## VALIDATION OUTCOME (Post-Hoc Review Mode)

When used for post-hoc validation of user modifications:

- ALL checks pass → proceed to commit
- Schema failure (invalid category, missing fields) → auto-correct trivial issues, then commit
- Content inaccuracy detected → auto-correct if high confidence, otherwise flag to user:
  - Category mismatch → auto-correct to the right category, show correction to user
  - Vague error_summary → auto-improve, show improvement to user
  - Overly generic proposed rule → flag to user, suggest more specific wording
- Soft warning (minor imprecision that doesn't affect rule quality) → note in output, proceed
